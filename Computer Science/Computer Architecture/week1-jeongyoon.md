

## **[COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.**

답변: CPU는 혼자 일을 하는 매개체가 아니기 때문에 ALU, 제어장치, 레지스터라는 팀을 구성합니다. 그중 ALU는 산술 논리 장치로서 계산과 판단을 맡고 있습니다. 

ALU가 하는 대표적인 일은 세가지로 산술연산, 논리연산, 그리고 비교입니다. 또한 계산 결과만을 내보내지 않고 플래그를 붙혀 계산 결과의 상태도 함께 넘겨줍니다.

레지스터는 ALU가 계산을 하기 위한 재료를 수급해주는 곳으로 매번 ALU가 RAM까지 값을 가지러 가는 것이 힘들기 때문에 CPU 안에서 아주 가까운 임시 저장소로서 존재합니다. 레지스터는 용량은 작지만 매우 빠릅니다.

ALU와 레지스터가 있어도 누군가 ‘지금은 무엇을 해야 하는지’ 알려 주지 않으면 CPU는 제대로 움직일 수가 없습니다. 이 역할을 해주는 것이 제어 장치입니다. 제어장치는 명령어의 opcode(연산 코드)를 읽고 CPU 안의 여러 회로에 신호를 보냅니다. 덧셈은 ALU가 진행하고 제어장치는 ALU에게 ‘이번에는 더하기를 해’라고 보내는 것과 같습니다.

컴퓨터의 머리는 생각을 흉내내는 무언가가 아니기 때문에 0과 1의 신호가 정해진 길을 따라 이동하고 제어장치가 그 길을 골라내고 ALU가 이를 계산하고 레지스터가 값을 붙잡아 두는 구조입니다. 작은 부품들이 팀을 이루는 순간 이는 CPU가 됩니다.

## **[COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.**

답변: 컴퓨터에서 실행되는 모든 프로그램들은 결국 CPU가 해석을 하고 이를 기계어 명령어들로 실행하는 연속성이 존재합니다. CPU는 이 명령어들을 하나하나 처리하며 프로그램을 수행하는데 이 일련의 과정을 명령어 사이클, fetch-decode-execute 사이클이라고 합니다. 

첫 번째, fetch는 명령어를 가져오는 것입니다. 명령어는 일반적으로 RAM에 저장되어 있는데 CPU 내부에는 PC라는 레지스터가 존재하며, 이 안에는 다음에 가져올 명령어의 주소를 저장하고 있습니다. PC가 가리키는 주소에서 명령어를 읽어와 명령어 레지스터(IR)에 저장합니다. fetch가 끝나면 PC는 다음 명령어 주소를 가리키도록 자동으로 증가합니다.

Decode는 말 그래도 명령어를 해독하는 것입니다. 명령어는 연산 코드, 오퍼랜드, 주소모드의 필드를 포함하며 제어 유닛은 이 정보를 해석해 CPU의 내부 회로를 설정하고 다음 단계인 실행을 준비합니다.

마지막으로 Execute는 실행 단계로 해석된 명령을 실제로 실행합니다. 연산이라면 ALU가 계산을 하고 메모리 접근이면 데이터를 읽거나 쓰게 됩니다. 이러한 결과들은 전부 레지스터에 저장하거나 메모리로 전송됩니다.

## **[COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.**

답변: 프로그램 속 소스코드는 compile, link, load의 순서로 메모리에 올려져 실행이 됩니다.

특정 프로그래밍 언어로 쓰여 있는 코드를 어셈블리어나 기계어 등의 low-level language로 번역해주는 프로그램을 컴파일러라고 합니다. 번역하는 과정이 컴파일이며 컴파일 전의 원래의 코드를 소스코드라 칭하고 번역이 된 코드를 목적 코드라고 부릅니다. 컴파일러는 컴파일 과정에서 프로그램의 뜻을 보존해야 하고 입력으로 들어온 소스코드를 목적코드로 변환 시킬 때 실용적인 측면에서 개선해야 하는 조건이 있습니다.

컴파일의 결과물로 나온 목적 코드들과 데이터, 라이브러리 등을 엮어서 실행 가능한 실행 파일로 만드는 것을 Link라고 합니다. 만약 system.out.println()이라는 함수가 소스코드에 존재한다고 하면 해당 함수를 CPU에서 읽었을 때 무엇인지 알 길이 없으므로 해당 함수에 대한 정보를 가지고 있는 다른 프로그램의 라이브러리에서 정보를 얻어야만 의미를 알 수 있습니다.

링커가 만든 실행 파일을 메모리에 올리는 것을 load, 이를 수행하는 프로그램을 로더라고 합니다. 로더는 프로그램의 실행을 위해 메모리의 공간을 확보하는 할당, 링킹, 메모리의 한계가 있을 때 발생하는 재배치, 프로그램을 할당된 메모리에 실제로 옮기는 적재 라는 4단계를 걸쳐 로드를 수행합니다.

## **[COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.**

답변: 프로그램의 소스코드 전체를 스캔하여 이를 목적 코드로 변환을 하고나서 실행 하는 방식이 컴파일 언어이고 우선 바로 실행하여 소스 코드의 한줄 한줄씩을 그자리에서 바로 기계어로 변환하면서 실행하는 방식을 인터프리터 언어라고 합니다. 

컴파일러 언어는 프로그램이 실행되기 전까지 초기 작업에 시간과 자원을 많이 투자해야하는 단점이 있지만 실행할 때는 인터프리터보다 빠른 속도를 가지는 장점이 있습니다.

인터프리터 언어는 반대로 실행 속도는 컴파일러를 사용하는 방식에 비해 느릴 수 밖에 없지만 에러를 한줄 단위로 바로바로 알려주어 실시간으로 코드 수정이 가능하단 점에서 개발 편의성이 높습니다.

## **[COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.**

답변: JIT 컴파일러는 실행 시점에서 프로그램 코드를 분석하고 자주 실행되는 부분을 찾아 해당 부분만 기계어로 변환하는 방식이다. 우리가 아는 컴파일러처럼 실행 전에 미리 컴파일을 하는 것이 아니라 실행 중에 실시간으로 컴파일이 이뤄진다. 즉, 인터프리터처럼 코드를 한 줄씩 읽으며 실행하지만 성능을 높이기 위해 자주 실행되는 부분은 컴파일해서 빠르게 실행할 수 있도록 하는 것이다.

장점은 당연히 핫 코드를 미리 컴파일하여 빠른 실행 속도를 가진 다는 점이다. 단점은 컴파일된 코드를 메모리에 저장하고 캐싱하기 때문에 인터프리터 방식보다 더 많은 메모리를 소비한다.

## **[COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.**

답변: 프로그램이 복잡해지면서 프로세스 하나 만을 사용해서 프로그램을 실행하기에는 어려워졌습니다. 따라서 프로그램 하나가 단순히 작업 하나를 진행하는 경우는 거의 없어졌습니다.

그래서 프로세스보다는 더 작은 실행 단위 개념이 필요해졌고 스레드라는 개념이 탄생했습니다. 프로세스가 메모리에 올라갈 때 운영체제로부터 시스템 자원을 할당받는데 프로세스마다 독립적인 메모리 영역을 주기 때문에 프로세스는 다른 프로세스의 변수나 자료에 접근 할 수 없습니다. 이와 다르게 스레드는 공유자원이라는 것이 존재해 별도의 스택을 가지고 있지만 힙 메모리는 서로 읽고 쓸 수 있습니다.

그렇기 때문에 프로세스는 오류가 발생해서 종료가 된다면 아무런 영향도 줄 수 없지만 스레드는 메모리 영역의 내용을 공유하기 때문에 하나의 스레드에서 오류가 발생한다면 프로세스 내의 다른 스레드에도 모두 영향이 가게 됩니다.

## **[COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.**

답변: PCB는 프로세스 제어 블록으로 운영체제가 프로세스 스케쥴링을 위해 프로세스에 관한 모든 정보를 가지고 있는 블록입니다. 프로세스가 생성될 때마다 공유의 PCB가 생성되고 프로세스가 완료되면 PCB는 제거됩니다.

프로세스는 CPU를 점유하고 작업을 처리하다가도 상태가 전이되면 작업 내용들을 모두 멈추고 CPU를 반환해야 합니다. 이때 진행하던 작업들을 저장하지 않으면 몽땅 날리고 어떤 작업을 해야하는 지 모르게 되는데, 이를 위해 PCB가 필요한 것입니다. 운영체제가 프로세스를 식별할 수 있도록 프로세스에 관한 모든 정보가 필요하기 때문입니다.

## **[COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.**

## **[COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.**

답변: cpu가 실행 중인 프로세스를 강제로 중단하고 다른 프로세스로 갈아타게 만드는 걸 담당하는 것이 바로 인터럽트입니다.

인터럽트가 발생하게 되면 CPU는 명령어 실행을 중단하고 현재 상태를 스택이나 레지스터에 저장합니다. 인터럽트의 종류를 확인하고 인터럽트 벡터를 참조해 해당 처리 프로그램인 ISR로 이동합니다. 실제 인터럽트 처리 코드를 수행하고 ISR 실행이 끝나면 저장해둔 프로그램 상태를 다시 복구하여 중단되었던 작업을 계속 진행합니다.

운영체제는 인터럽트가 발생하면 하던 프로세스를 멈추고 커널 모드로 전환하여 작업을 처리합니다. 중요한 인터럽트가 발생하게 되면 OS는 현재 프로세스의 상태를 저장하고 ISR을 실행한 뒤, 필요에 따라 다른 프로세스를 CPU에 할당하여 실행합니다.

## **[COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.**

답변: 프로세스의 상태는 5가지의 주요 상태인 생성, 준비, 실행, 대기/블로킹, 완료로 이뤄집니다.

생성은 프로세스가 메모리를 할당받고 PCB가 생성되어 실행 준비를 완료한 상태입니다.

준비는 생성된 프로세스가 CPU를 얻을 때 까지 기다리는 상태입니다. 실행은 CPU에서 프로세스가 실행 중인 상태로 CPU의 개수만큼 프로세스가 실행 상태가 됩니다. 이때 입출력 요청에 의해 해당 I/O 작업이 끝날 때까지 대기하는 상태를 대기 상태라고 하고 CPU 스케쥴러가 대기 상태에 있는 프로세스 중 하나를 가져다 실행 상태로 만듭니다.

프로세스가 정상 혹은 강제로 종료되어 자신이 가진 메모리를 반환하고 PCB가 삭제된 상태를 완료라고 합니다.

## **[COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.**

답변: 컴퓨터가 멀티태스킹을 할 수 있는 핵심적인 이유는 CPU가 처리해야 할 여러 작업의 실행 시간을 아주 잘게 쪼개어 번갈아 가며 실행하기 때문입니다. 이 과정을 가능하게끔 하는 것이 CPU 스케쥴링과 컨텍스트 스위칭입니다.

CPU 스케줄링은 실행 중인 프로세스가 일정 시간을 다 쓰거나 입출력 대기 상태가 되면 스케줄러가 다음 실행할 작업을 선택하며 CPU의 자원을 효율적으로 배분합니다.

컨텍스트 스위칭은 CPU가 실행하던 작업을 멈추고 다른 작업으로 전환 할 때 기존 작업의 상태를 저장하고 새로운 작업의 이전 상태를 복원해줍니다. 이 두가지의 역할을 통해 멀티태스킹을 가능케 합니다.
