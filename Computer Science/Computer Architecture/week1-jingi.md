## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변:

ALU는 Arthimetic Logic Unit으로, 말 그대로 CPU에서 산술, 논리 등의 연산을 담당하는 수행하는 구성 요소입니다.

제어장치는 명령어를 해석하고, CPU의 전체적인 실행을 위해 구성 요소 간에 제어 신호를 보내는 역할을 합니다.

레지스터는 CPU 내부에 존재하는 작은 데이터 저장 공간입니다.

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변: 

CPU가 명령어를 실행하는 과정은 Fetch, Decode, Execute 단계로 나눌 수 있습니다.

먼저 Fetch 단계는 PC 즉, Program Counter가 가리키는 다음 명령어를 가져오는 단계입니다.

Decode 단계는 제어 장치를 통해 Fetch 단계에서 가져온 명령어가 무엇인지 해석하는 단계입니다.

마지막으로 Execute 단계는 해석한 명령어를 실제로 수행하는 단계입니다.

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변: 

소스 코드가 프로그램으로 실행되는 과정은 컴파일, 링킹, 로딩, 실행 단계로 설명할 수 있습니다.

먼저 컴파일 단계는 사람이 작성한 소스 코드를 기계가 해석할 수 있는 언어로 번역하는 작업으로, 이때 언어로는 대표적으로 어셈블리 언어 및 기계어가 있습니다.

링킹 단계는 컴파일된 소스 코드가 사용하는 외부 라이브러리 등을 탐색하고, 이를 심볼릭으로 연결하여 하나의 오브젝트 파일을 생성하는 단계입니다.

로딩 단계는 생성된 오브젝트 파일을 실행하기 위해 이를 메모리에 적재하는 단계입니다.

마지막으로 실행 단계는 메모리에 적재된 파일이 실제 프로세스로 실행되는 단계입니다.

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변:

컴파일 언어와 인터프리터 언어는 대표적으로 프로그래밍 언어가 프로그램으로 변환되는 방식을 구분한 것입니다.

먼저 컴파일 언어는 프로그래밍 언어를 전체적으로 실행 가능한 형태로 변환하고, 이를 실행하는 방식으로 설계된 언어를 의미합니다.

인터프리터 언어는 프로그래밍 언어를 실행하는 과정에서 소스코드가 실행될 수 있도록 해석하는 언어를 의미합니다.

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변:

프로세스는 실행 상태에 따라 다양한 상태로 구분할 수 있습니다.

먼저 Ready 상태는 프로세스가 메모리에 적재되어있지만 아직 CPU 사용을 할당받지 못하는 상태입니다.

이후 Running 상태는 프로세스가 디스패치 되어 CPU 사용을 할당받은 상태입니다.

이후 Waiting 상태는 프로세스가 입출력 처리와 같은 작업이 완료되는 것을 기다리는 상태입니다.

마지막으로 Terminated 상태는 프로세스의 실행이 모두 종료된 상태를 의미합니다.

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변:

CPU 스케줄러는 여러 프로세스에게 적절한 CPU 사용량을 분배하기 위해 필요한 요소입니다.

스케줄러가 없다면 하나의 프로세스가 CPU 사용량을 지나치게 점유할 수 있어 다른 프로세스의 실행이 지연되는 문제가 발생할 수 있습니다.

대표적인 스케줄링 알고리즘으로는 FCFS, SJF, Round Robin, SRT, Priority Scheduling 등이 있습니다.

FCFS 알고리즘은 First Come, First Served의 약자로서 Ready Queue에 도착한 프로세스의 순서대로 CPU 사용 권한을 부여하는 알고리즘입니다.

SJF 알고리즘은 Shortest Job First의 약자로서 남은 실행 시간이 가장 짧은 프로세스부터 CPU 사용 권한을 부여하는 알고리즘입니다.

Round Robin 알고리즘은 Time Quantum이라는 CPU 사용 시간을 결정한 뒤, 프로세스가 이 CPU 사용 시간을 번갈아가며 균등하게 사용하는 알고리즘입니다.

SRT는 Shortest Remaining Time의 약자로서 남은 실행 시간이 가장 적은 프로세스부터 CPU 사용 권한을 갖는 알고리즘입니다. 이는 기본적으로 SJF 알고리즘과 유사하지만, SJF 알고리즘은 비선점형 스케줄링인 반면, SRT 알고리즘은 남은 실행 시간이 더 짧은 프로세스가 대기 중일 경우, 이 프로세스가 CPU 사용을 가로챌 수 있는 선점형 스케줄링 알고리즘이라는 부분에서 차이가 있습니다.

Priority Scheduling은 각 프로세스마다 우선순위 값을 정하여 우선순위가 높은 프로세스부터 CPU 사용 권한을 부여하는 알고리즘입니다.

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

답변:

스케줄링 알고리즘은 프로세스가 실행 되는 중간에 다른 프로세스가 CPU 사용 권한을 가로챌 수 있는지 여부에 따라 선점형 스케줄링과 비선점형 스케줄링으로 구분할 수 있습니다.

먼저 선점형 스케줄링은 프로세스가 실행되는 중간에 알고리즘에 따라 다른 프로세스가 CPU를 사용할 수 있는 권한이 생기면, 운영체제가 중간에 CPU 권한을 회수해서 실행 중인 프로세스를 중간에 중지시키고 다른 프로세스가 CPU 사용 권한을 가로채는 알고리즘입니다. 대표적으로 라운드 로빈 그리고 SJF 알고리즘이 있습니다.

비선점형 스케줄링은 대기 중인 다른 프로세스가 알고리즘에 따라 CPU 사용 권한이 생기더라도, 현재 실행 중인 프로세스가 있다면 이 프로세스의 실행이 종료된 후에 CPU를 사용할 수 있는 알고리즘입니다. 대표적으로 FCFS, SJF, Priority Scheduling, HRN 등의 알고리즘이 있습니다.

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변:

Time Quantum은 한 프로세스가 CPU를 사용하기 시작할 때, CPU를 사용할 수 있는 제한시간을 의미합니다.

Time Quantum이 길 경우, 프로세스의 CPU 할당을 교체하는 컨텍스트 스위칭이 적게 일어나기 때문에 그만큼의 지연 시간이 감소하지만, 하나의 프로세스가 CPU를 차지하는 시간이 늘어나기 때문에 각 프로세스의 지연 시간이 늘어난다는 단점이 있습니다.

반면에 Time Quantum이 너무 짧을 경우, 여러 프로세스가 짧은 간격으로 CPU를 할당받아 각 프로세스에 대한 지연 시간이 감소하지만, 컨텍스트 스위칭의 빈도수가 늘어나는 만큼 지연시간이 늘어난다는 단점이 있습니다.

## [COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.

답변:

멀티태스킹이 가능한 이유는 여러 프로세스가 매우 짧은 시간동안 CPU 자원을 번갈아가면서 사용할 수 있기 때문입니다. 이 과정에서 사용되는 것이 CPU 스케줄링과 컨텍스트 스위칭인데, CPU 스케줄링을 통해 Ready Queue에서 다음 실행할 적절한 프로세스를 선택하고, 컨텍스트 스위칭을 통해 기존에 실행 중인 프로세스를 다시 준비 상태로 저장하고, 새로운 프로세스의 진행 상황을 복원하여 이를 실행하는 작업을 통해 여러 프로세스가 CPU 자원을 번갈아가면서 사용합니다. 따라서 실제로 여러 프로세스가 동시에 실행되는 것은 아니지만, 매우 짧은 시간마다 CPU를 번갈아가면서 사용하기 때문에 여러 프로그램이 동시에 실행되는 것처럼 보여 멀티태스킹이 가능합니다.
