## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변:

ALU는 Arthimetic Logic Unit으로, 말 그대로 CPU에서 산술, 논리 등의 연산을 담당하는 수행하는 구성 요소입니다.

제어장치는 명령어를 해석하고, CPU의 전체적인 실행을 위해 구성 요소 간에 제어 신호를 보내는 역할을 합니다.

레지스터는 CPU 내부에 존재하는 작은 데이터 저장 공간입니다.

참고 자료:

- ALU 설명이 정확하지만 문장이 짧게 끝남 → 산술연산(덧셈 등)과 논리연산(AND/OR/시프트 등) 예시를 하나씩 붙이면 설득력이 올라감
- 제어장치의 "제어 신호"가 구체적으로 어떤 동작을 지시하는지(메모리 읽기/쓰기 타이밍, ALU 연산 종류 선택 등) 한 가지 사례를 덧붙이면 좋음
- 레지스터를 "작은 저장 공간"이라고만 설명했는데, 왜 메모리보다 빠른지(CPU 내부에 물리적으로 위치, 캐시보다도 빠른 계층)와 대표 레지스터 종류(PC, IR, 범용 레지스터 등)를 언급하면 깊이가 생김
- 마무리에 세 요소가 함께 어떻게 협력하는지(명령어 사이클과 연결) 한 문장을 추가하면 답변이 유기적으로 느껴짐

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변: 

CPU가 명령어를 실행하는 과정은 Fetch, Decode, Execute 단계로 나눌 수 있습니다.

먼저 Fetch 단계는 PC 즉, Program Counter가 가리키는 다음 명령어를 가져오는 단계입니다.

Decode 단계는 제어 장치를 통해 Fetch 단계에서 가져온 명령어가 무엇인지 해석하는 단계입니다.

마지막으로 Execute 단계는 해석한 명령어를 실제로 수행하는 단계입니다.

참고 자료:

- 정확하고 간결하게 잘 정리된 답변
- 꼬리질문 대비: PC가 정확히 언제 증가하는지(Fetch 직후인지), Execute 이후 결과를 메모리/레지스터에 반영하는 Store(Write-back) 단계까지 언급할 수 있는지 미리 생각해두면 좋음
- 인터럽트 확인이 사이클의 어느 시점에 이루어지는지도 자주 나오는 꼬리질문이므로 준비 필요

보충 개념:
- PC 증가 시점: Fetch 직후 PC는 다음 명령어를 가리키도록 곧바로 증가함(Execute 결과와 무관하게 미리 갱신) — 단, 분기(jump/branch) 명령어의 경우 Execute 단계에서 PC가 다시 갱신됨
- Store(Write-back) 단계: Execute에서 계산된 결과를 레지스터나 메모리에 반영하는 단계를 별도로 부르기도 함(교재에 따라 Execute에 포함시키기도 함)
- 인터럽트 확인 시점: 보통 한 명령어 사이클이 완전히 끝난 직후(Execute 종료 시점)에 인터럽트 발생 여부를 확인하고, 발생했다면 다음 Fetch 대신 인터럽트 처리 루틴으로 진입

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변: 

소스 코드가 프로그램으로 실행되는 과정은 컴파일, 링킹, 로딩, 실행 단계로 설명할 수 있습니다.

먼저 컴파일 단계는 사람이 작성한 소스 코드를 기계가 해석할 수 있는 언어로 번역하는 작업으로, 이때 언어로는 대표적으로 어셈블리 언어 및 기계어가 있습니다.

링킹 단계는 컴파일된 소스 코드가 사용하는 외부 라이브러리 등을 탐색하고, 이를 심볼릭으로 연결하여 하나의 오브젝트 파일을 생성하는 단계입니다.

로딩 단계는 생성된 오브젝트 파일을 실행하기 위해 이를 메모리에 적재하는 단계입니다.

마지막으로 실행 단계는 메모리에 적재된 파일이 실제 프로세스로 실행되는 단계입니다.

참고 자료:

- 개념 정정 필요: "링킹 단계에서 오브젝트 파일을 생성한다"고 했는데, 오브젝트 파일은 컴파일(어셈블) 단계의 결과물이고 링킹의 결과물은 "실행 파일(executable)"임 → 이 부분 표현을 바꿔야 함
- "컴파일 단계에서 어셈블리 언어 및 기계어로 변환"이라 했는데, 엄밀히는 컴파일러가 소스코드를 어셈블리 코드로 바꾸고, 어셈블러가 이를 기계어(오브젝트 파일)로 바꾸는 별도 단계로 나뉨 → 필요하면 이 구분을 언급하면 좋음
- 로딩/실행 단계 설명 자체는 적절함
- 전체 흐름을 다시 말할 때는 "컴파일러가 오브젝트 파일을 만들고 → 링커가 여러 오브젝트 파일+라이브러리를 묶어 실행 파일을 만들고 → 로더가 이를 메모리에 올린다"는 순서로 다시 정리해보면 좋음

보충 개념:
- 산출물 정리: 컴파일러 결과물 = 어셈블리 코드(.s) → 어셈블러 결과물 = 목적/오브젝트 파일(.o, 기계어이지만 심볼 미해석 상태) → 링커 결과물 = 실행 파일(여러 오브젝트 파일·라이브러리의 심볼을 모두 해석해 하나로 합침)
- 링커가 하는 일: 여러 오브젝트 파일에 흩어진 함수·변수 심볼의 실제 주소를 찾아 연결(symbol resolution), 코드/데이터 섹션을 하나의 주소 공간으로 재배치(relocation)
- 로더가 하는 일: 실행 파일을 메모리에 적재하고, 필요하면 동적 라이브러리를 연결하며, 초기 PC 등을 설정해 실행할 준비를 마침

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변:

컴파일 언어와 인터프리터 언어는 대표적으로 프로그래밍 언어가 프로그램으로 변환되는 방식을 구분한 것입니다.

먼저 컴파일 언어는 프로그래밍 언어를 전체적으로 실행 가능한 형태로 변환하고, 이를 실행하는 방식으로 설계된 언어를 의미합니다.

인터프리터 언어는 프로그래밍 언어를 실행하는 과정에서 소스코드가 실행될 수 있도록 해석하는 언어를 의미합니다.

참고 자료:

- 정의 자체는 맞지만 비교 관점(장단점)이 빠져 있어 다소 평면적인 답변으로 들림
- 추가하면 좋은 비교축: 실행 속도(컴파일이 보통 빠름), 이식성(인터프리터가 유리), 디버깅 편의성, 에러 발견 시점(컴파일타임 vs 런타임) 중 1~2개
- 실제 언어 예시(C/C++ vs Python/JavaScript)를 곁들이면 답변이 훨씬 구체적으로 들림
- 005(JIT)와 이어지는 질문이므로, 이 답변 끝에 "이 둘을 절충한 방식도 있다"는 식으로 연결하면 흐름이 자연스러움

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요. ⭐

답변:

JIT 컴파일은 Just-In-Time Compilation의 약자로, 인터프리터 방식과 컴파일 방식을 함께 사용하는 실행 방식을 의미합니다.

기본적으로 인터프리터 언어처럼 소스 코드를 한 줄씩 해석하며 실행하다가, 반복적으로 실행되는 코드 영역이 있으면 이 부분을 기계어로 컴파일하여 캐싱해두고, 이후에는 캐싱된 기계어를 바로 실행하는 방식으로 동작합니다.

JIT 컴파일의 장점은 인터프리터 방식보다 실행 속도가 빠르다는 점입니다. 반복되는 코드를 매번 새로 해석하지 않고, 이미 컴파일된 기계어를 재사용하기 때문입니다.

하지만 코드를 컴파일하는 과정 자체에서 추가적인 CPU 및 메모리 자원이 소모되고, 컴파일이 이루어지기 전까지는 인터프리터 방식으로 실행되기 때문에 초기 실행 속도는 느리다는 단점이 있습니다.

대표적으로 Java의 JVM이나 JavaScript 엔진인 V8이 JIT 컴파일 방식을 사용합니다.

참고 자료:

- JIT(Just-In-Time) = 인터프리터 방식과 컴파일 방식을 혼합한 실행 기법
- 기본 동작: 처음엔 인터프리터처럼 바이트코드를 한 줄씩 실행하다가, 반복적으로 실행되는(hot) 코드를 감지하면 해당 부분만 기계어로 컴파일해 캐싱하고, 이후에는 캐싱된 기계어를 바로 실행
- 장점: 순수 인터프리터보다 실행 속도가 빠름 / 런타임에 수집한 실제 실행 정보를 바탕으로 최적화 가능(정적 컴파일이 못 하는 부분도 최적화 가능)
- 단점: 초기 워밍업 시간이 필요(컴파일되기 전까지는 느림) / 컴파일 작업 자체가 추가 CPU·메모리 오버헤드를 유발
- 대표 사례: Java의 JVM(HotSpot), JavaScript 엔진(V8) 등
- 답변 구성 팁: "정의 → 동작 원리 → 장점 → 단점 → 예시" 순서로 말하면 자연스럽게 완결됨. 004(컴파일/인터프리터 차이)와 연결지어 "그 둘의 절충안"이라는 포지셔닝으로 시작하면 좋음

## [COMMON-006] 프로그램과 프로세스의 차이에 대해 설명해 주세요.

답변:

프로그램은 실행 파일이 디스크에 저장되어 있어 실행되지 않는 상태일 때를 의미합니다.

반면, 프로세스는 디스크에 저장되어 있는 프로그램이 메모리에 적재되어 실행 중인 상태일 때를 의미합니다.

참고 자료:

- 정의 자체는 정확함
- 다만 비교가 "저장 vs 실행"에서 그침 → 프로세스가 되면서 새로 갖게 되는 것(고유한 메모리 공간, PCB, CPU/메모리 등 자원 할당)을 한 문장 덧붙이면 한 단계 더 깊은 답변이 됨
- 008(PCB) 질문과 자연스럽게 연결되는 포인트이니 함께 준비하면 좋음

보충 개념:
- 프로세스가 되면서 새로 갖는 것: 고유한 PID, 독립된 가상 주소 공간(Code/Data/Heap/Stack), PCB(상태·레지스터 값 등 관리 정보), OS로부터 할당받는 CPU 시간·메모리 등의 자원
- 대비 포인트: 프로그램은 정적(static)이고 수동적인 코드 덩어리인 반면, 프로세스는 동적(dynamic)이며 New/Ready/Running 같은 상태를 가진다는 점으로 마무리하면 좋음

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

답변:

프로세스는 하나의 프로그램이 메모리에 적재되어 실행되는 단위이고, 스레드는 프로세스 내의 실행 흐름을 의미합니다.

메모리의 영역 중, 프로세스 내의 여러 스레드는 Data, Heap, Code 영역을 서로 공유하고, 독립적인 Stack, PC, Register 영역을 갖습니다.

참고 자료:

- 공유/비공유 메모리 영역을 정확히 짚은 좋은 답변
- 보강 포인트: 프로세스 간 통신은 IPC가 필요해 비용이 큰 반면, 스레드는 메모리를 공유해 통신이 가볍다는 비교를 추가하면 실무적 이해도를 보여줄 수 있음
- 안정성 측면(스레드 하나의 오류가 같은 프로세스의 다른 스레드에도 영향을 줄 수 있다는 점)도 언급하면 답변이 풍부해짐

보충 개념:
- 프로세스 간 통신(IPC) 예시: 파이프, 소켓, 공유 메모리, 메시지 큐 등 — 커널 개입이 필요해 상대적으로 비용이 큼
- 스레드 간 통신: 같은 프로세스의 메모리(Heap·전역 변수 등)를 그대로 공유하므로 커널 개입 없이 빠르게 데이터 교환 가능(다만 동기화 문제가 생겨 뮤텍스·세마포어 같은 장치가 필요)
- 안정성 비교: 한 프로세스의 크래시는 독립된 주소 공간 덕분에 다른 프로세스에 영향 없음 / 한 스레드의 크래시(잘못된 메모리 접근 등)는 같은 프로세스 내 다른 스레드에도 영향을 줄 수 있음

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요. ⭐

답변:

PCB는 Process Control Block의 약자로, 운영체제가 하나의 프로세스를 관리하기 위해 유지하는 자료구조를 의미합니다.

PCB에는 프로세스의 상태, PID, PC 값, 레지스터 값, 스케줄링을 위한 우선순위 정보 등 프로세스를 실행하고 관리하는 데 필요한 정보들이 저장되어 있습니다.

운영체제가 PCB를 사용하는 이유는, 여러 프로세스를 동시에 관리해야 하는 상황에서 각 프로세스의 실행 상태를 구분하고 추적하기 위함입니다.

또한 컨텍스트 스위칭이 발생할 때, 현재 실행 중이던 프로세스의 상태를 PCB에 저장해두었다가 이후 해당 프로세스가 다시 CPU를 할당받을 때 PCB에 저장된 정보를 그대로 복원하여 실행을 이어갈 수 있도록 하기 위해 PCB가 필요합니다.

참고 자료:

- PCB(Process Control Block) = 프로세스 하나를 관리하기 위해 커널이 유지하는 자료구조

- 정의와 발생 시점 설명 자체는 정확함

- 오버헤드 원인: 레지스터 저장/복원 자체의 시간 비용 + 캐시·TLB가 이전 프로세스 기준으로 채워져 있어 전환 직후 캐시 미스가 늘어나는 간접 비용
- 발생 시점 보강: 타임 슬라이스 만료, I/O 요청 외에도 "더 높은 우선순위 프로세스 등장", "인터럽트 발생" 시에도 발생 가능

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요. ⭐

답변: 

프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭은 전환 시 교체해야 하는 정보의 범위에서 차이가 있습니다.

프로세스 컨텍스트 스위칭은 서로 다른 프로세스 간에 전환이 이루어지므로, 각 프로세스가 독립적으로 갖고 있는 주소 공간, 즉 페이지 테이블까지 함께 교체해야 합니다.

반면 스레드 컨텍스트 스위칭은 같은 프로세스에 속한 스레드 간의 전환이기 때문에, Code, Data, Heap 영역과 같은 주소 공간을 그대로 공유합니다. 따라서 레지스터 값이나 스택 포인터와 같은 최소한의 정보만 교체하면 되기 때문에, 프로세스 컨텍스트 스위칭에 비해 오버헤드가 작다는 차이가 있습니다.

참고 자료: 

- 프로세스 컨텍스트 스위칭: 서로 다른 프로세스 간 전환이므로 주소 공간(메모리 매핑, 페이지 테이블) 자체가 바뀜 → TLB flush, 캐시 미스 등으로 오버헤드가 큼
- 스레드 컨텍스트 스위칭(같은 프로세스 내 스레드 간 전환): 주소 공간을 공유하므로 페이지 테이블 교체가 필요 없고, 레지스터 값·스택 포인터 등만 교체 → 상대적으로 가벼움
- 핵심 논리 흐름: "무엇을 저장/복원해야 하는가(주소 공간 포함 여부)"의 차이가 오버헤드 차이로 이어진다는 식으로 답변 구성
- 007(프로세스 vs 스레드의 메모리 공유 구조) 답변과 자연스럽게 연결지어 설명하면 좋음

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변:

운영체제에서 유저 모드와 커널 모드를 구분하는 이유는, 무분별한 권한으로 커널 레벨에 접근하여 직접적으로 컴퓨팅 리소스를 제어하는 것을 방지하기 위함입니다.

기본적으로 유저 모드는 운영체제 위에서 운영되는 애플리케이션이 실행되는 제한된 모드입니다.

반면, 커널 모드는 운영체제의 커널에 접근하여 메모리, 디스크 등의 자원을 직접적으로 제어할 수 있는 권한을 갖고 있습니다.

이러한 접근 범위를 유저 모드와 커널 모드로 구분함으로써, 하나의 프로그램이 무분별하게 커널에 접근하여 시스템 전체에 영향을 주는 것을 방지합니다.

참고 자료:

- 이유, 각 모드의 권한 범위, 결론까지 논리적으로 잘 짜여진 답변
- 보강 포인트: 유저 모드에서 커널 기능이 필요할 때는 시스템 콜을 통해 커널 모드로 전환한다는 연결 고리를 한 문장 추가하면 012·013번 질문과 자연스럽게 이어지는 답변이 됨

## [COMMON-012] 시스템 콜이 무엇이고, 사용자 프로그램이 시스템 콜을 호출하는 이유를 설명해 주세요. ⭐

답변: 

시스템 콜은 유저 프로그램이 운영체제의 커널이 제공하는 기능을 사용하기 위해 호출하는 인터페이스를 의미합니다.

사용자 프로그램이 시스템 콜을 호출하는 이유는, 파일 입출력이나 프로세스 생성과 같이 하드웨어 자원이 필요한 작업은 유저 모드에서 직접 수행할 수 없고, 커널 모드의 권한이 필요하기 때문입니다.

즉, 유저 프로그램은 이러한 자원을 직접 제어하지 않고 시스템 콜을 통해 커널에게 필요한 작업을 대신 요청하고, 커널이 이를 처리한 결과를 반환하는 방식으로 동작합니다.

참고 자료:

- 시스템 콜 = 유저 프로그램이 커널이 제공하는 기능(파일 입출력, 프로세스 생성/종료, 메모리 할당 등)을 요청하기 위한 인터페이스
- 호출 이유: 디스크, 메모리, 네트워크 같은 하드웨어 자원은 보안·안정성을 위해 커널만 직접 제어 가능 → 유저 프로그램이 이 자원을 쓰려면 반드시 커널에 "요청"해야 함
- 대표 예시: read(), write(), fork(), exec(), open() 등
- 011(유저/커널 모드) 답변과 연결지어 "왜 직접 접근하지 못하고 시스템 콜을 거쳐야 하는지"를 설명하면 답변에 일관성이 생김

## [COMMON-013] 시스템 콜이 실행될 때 유저 모드에서 커널 모드로 전환되는 과정을 설명해 주세요. ⭐

답변:

시스템 콜이 호출되면, 유저 프로그램은 먼저 호출하려는 시스템 콜의 번호와 필요한 인자들을 정해진 레지스터에 저장합니다.

이후 trap이라는 소프트웨어 인터럽트 명령을 실행하게 되는데, 이 명령이 실행되면 유저 모드에서 커널 모드로 전환됩니다.

모드가 전환되면 현재까지의 실행 과정을 저장한 뒤, 해당 시스템 콜을 처리하는 커널의 핸들러 코드로 이동합니다.

커널은 요청받은 작업을 처리한 뒤 그 결과를 반환하고, 다시 유저 모드로 전환되어 원래 실행하고 있던 지점으로 복귀합니다.

참고 자료:

- 시스템 콜 호출 시 트랩(trap)/소프트웨어 인터럽트 명령(예: x86의 syscall, int 0x80) 실행
- 전환 과정 순서: ① 유저 프로그램이 시스템 콜 번호와 인자를 레지스터에 설정 ② trap 명령 실행 → CPU가 모드 비트를 커널 모드로 전환 ③ 현재 실행 문맥(레지스터 등) 저장 ④ 시스템 콜 벡터 테이블을 참조해 해당 커널 핸들러로 점프 ⑤ 커널이 요청 처리 ⑥ 처리 결과 반환 후 다시 유저 모드로 복귀하여 원래 실행 지점 재개
- 015(HW/SW 인터럽트) 답변과 연결지어, 시스템 콜은 "소프트웨어 인터럽트"의 한 종류라는 점을 언급하면 좋음

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요. ⭐

답변:

인터럽트는 CPU 외부 또는 내부에서 발생한 이벤트를 CPU에게 알려, 현재 실행 중인 작업을 일시적으로 중단시키는 신호를 의미합니다.

인터럽트가 발생하면 CPU는 현재 실행 중이던 명령어 사이클을 마친 뒤 인터럽트 발생 여부를 확인합니다. 인터럽트가 발생했다면, 현재까지의 실행 상태를 저장합니다.

이후 CPU는 인터럽트 벡터 테이블을 참조하여 해당 인터럽트에 대응하는 ISR, 즉 Interrupt Service Routine으로 이동하여 이를 실행합니다.

운영체제는 이 과정에서 인터럽트의 종류에 맞는 코드를 실행하고, 처리가 끝나면 이전에 저장해둔 상태를 복원하여 중단되었던 작업을 이어서 실행합니다.

참고 자료:

- 인터럽트 = CPU 외부(하드웨어) 또는 내부(소프트웨어)에서 발생한 이벤트를 CPU에게 알려, 현재 실행 중인 작업을 잠시 멈추게 하는 신호
- 처리 흐름: ① 현재 실행 중인 명령어 사이클 완료 ② 인터럽트 발생 여부 확인 ③ 현재 실행 상태(PC, 레지스터 등) 저장 ④ 인터럽트 벡터 테이블을 참조해 해당 ISR(Interrupt Service Routine) 실행 ⑤ 처리 완료 후 저장해둔 상태를 복원하여 원래 작업 재개
- 운영체제 관점: 인터럽트 발생 시 필요하면 해당 이벤트에 맞는 커널 코드(디바이스 드라이버 등)를 실행하고, 스케줄링이 필요한 상황이면 컨텍스트 스위칭으로 이어질 수 있음
- 002(명령어 사이클), 009(컨텍스트 스위칭)와 연결지어 설명하면 답변에 깊이가 생김

## [COMMON-015] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이에 대해 설명해 주세요. ⭐

답변:

하드웨어 인터럽트와 소프트웨어 인터럽트는 인터럽트를 발생시키는 주체가 무엇인지에 따라 구분됩니다.

하드웨어 인터럽트는 키보드, 마우스, 타이머, 디스크와 같은 외부 장치가 발생시키는 인터럽트로, CPU가 실행하고 있는 명령어 흐름과 무관하게 비동기적으로 발생한다는 특징이 있습니다.

반면 소프트웨어 인터럽트는 실행 중인 프로그램이 trap과 같은 특정 명령어를 통해 의도적으로 발생시키는 인터럽트로, 대표적으로 시스템 콜이 있습니다. 이는 프로그램의 실행 흐름 안에서 예측 가능한 시점에 동기적으로 발생한다는 점에서 하드웨어 인터럽트와 차이가 있습니다.

참고 자료:

- 하드웨어 인터럽트: 외부 장치(키보드, 마우스, 타이머, 디스크 등)가 발생시키는 인터럽트, CPU의 명령어 실행 흐름과 비동기적으로 발생
- 소프트웨어 인터럽트: 프로그램이 특정 명령어(trap 등)를 실행해 의도적으로 발생시키는 인터럽트 (예: 시스템 콜, 0으로 나누기 같은 예외/exception)
- 비교 포인트: 발생 주체(외부 장치 vs 실행 중인 프로그램 자신), 발생 시점의 예측 가능성(비동기 vs 동기)
- 013(시스템 콜)과 연결지어, 시스템 콜이 소프트웨어 인터럽트의 대표적인 예라고 설명하면 답변이 자연스럽게 이어짐

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변:

프로세스는 실행 상태에 따라 다양한 상태로 구분할 수 있습니다.

먼저 Ready 상태는 프로세스가 메모리에 적재되어있지만 아직 CPU 사용을 할당받지 못하는 상태입니다.

이후 Running 상태는 프로세스가 디스패치 되어 CPU 사용을 할당받은 상태입니다.

이후 Waiting 상태는 프로세스가 입출력 처리와 같은 작업이 완료되는 것을 기다리는 상태입니다.

마지막으로 Terminated 상태는 프로세스의 실행이 모두 종료된 상태를 의미합니다.

참고 자료:

- Ready, Running, Waiting, Terminated 설명은 정확함
- 다만 "New(생성)" 상태가 빠져 있음 — 프로세스가 처음 생성되어 아직 Ready Queue에 들어가기 전 상태도 언급해야 다섯 가지 상태가 완전히 갖춰짐
- 각 상태 간 전이 조건(예: Running→Waiting은 I/O 요청 발생, Waiting→Ready는 I/O 완료)을 화살표 형태로 정리해서 말하면 더 체계적으로 들림

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변:

CPU 스케줄러는 여러 프로세스에게 적절한 CPU 사용량을 분배하기 위해 필요한 요소입니다.

스케줄러가 없다면 하나의 프로세스가 CPU 사용량을 지나치게 점유할 수 있어 다른 프로세스의 실행이 지연되는 문제가 발생할 수 있습니다.

대표적인 스케줄링 알고리즘으로는 FCFS, SJF, Round Robin, SRT, Priority Scheduling 등이 있습니다.

FCFS 알고리즘은 First Come, First Served의 약자로서 Ready Queue에 도착한 프로세스의 순서대로 CPU 사용 권한을 부여하는 알고리즘입니다.

SJF 알고리즘은 Shortest Job First의 약자로서 남은 실행 시간이 가장 짧은 프로세스부터 CPU 사용 권한을 부여하는 알고리즘입니다.

Round Robin 알고리즘은 Time Quantum이라는 CPU 사용 시간을 결정한 뒤, 프로세스가 이 CPU 사용 시간을 번갈아가며 균등하게 사용하는 알고리즘입니다.

SRT는 Shortest Remaining Time의 약자로서 남은 실행 시간이 가장 적은 프로세스부터 CPU 사용 권한을 갖는 알고리즘입니다. 이는 기본적으로 SJF 알고리즘과 유사하지만, SJF 알고리즘은 비선점형 스케줄링인 반면, SRT 알고리즘은 남은 실행 시간이 더 짧은 프로세스가 대기 중일 경우, 이 프로세스가 CPU 사용을 가로챌 수 있는 선점형 스케줄링 알고리즘이라는 부분에서 차이가 있습니다.

Priority Scheduling은 각 프로세스마다 우선순위 값을 정하여 우선순위가 높은 프로세스부터 CPU 사용 권한을 부여하는 알고리즘입니다.

참고 자료:

- 필요성, 각 알고리즘의 정의, 그리고 SJF와 SRT의 선점/비선점 차이까지 정확하게 짚은 매우 완성도 높은 답변
- 더 보강하려면: 스케줄링 알고리즘을 평가하는 기준(대기 시간, 응답 시간, 처리량, 공정성 등)을 한 문장 추가하면 "왜 여러 알고리즘이 존재하는지"까지 답변이 커버됨
- 이 답변에서 정리한 SJF=비선점형 / SRT=선점형 구분을 018번 답변에도 동일하게 적용해야 함 (아래 018 참고)

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

답변:

스케줄링 알고리즘은 프로세스가 실행 되는 중간에 다른 프로세스가 CPU 사용 권한을 가로챌 수 있는지 여부에 따라 선점형 스케줄링과 비선점형 스케줄링으로 구분할 수 있습니다.

먼저 선점형 스케줄링은 프로세스가 실행되는 중간에 알고리즘에 따라 다른 프로세스가 CPU를 사용할 수 있는 권한이 생기면, 운영체제가 중간에 CPU 권한을 회수해서 실행 중인 프로세스를 중간에 중지시키고 다른 프로세스가 CPU 사용 권한을 가로채는 알고리즘입니다. 대표적으로 라운드 로빈 그리고 SJF 알고리즘이 있습니다.

비선점형 스케줄링은 대기 중인 다른 프로세스가 알고리즘에 따라 CPU 사용 권한이 생기더라도, 현재 실행 중인 프로세스가 있다면 이 프로세스의 실행이 종료된 후에 CPU를 사용할 수 있는 알고리즘입니다. 대표적으로 FCFS, SJF, Priority Scheduling, HRN 등의 알고리즘이 있습니다.

참고 자료:

- 정의와 결론 자체는 명확하나, 예시 알고리즘 분류에 오류가 있음: 선점형 예시로 "라운드 로빈, SJF"를 들었는데, SJF는 비선점형 알고리즘임(바로 뒤 문단에서도 비선점형 예시로 SJF를 다시 언급해 스스로 모순됨) → 선점형의 올바른 예시는 SJF가 아니라 SRT(Shortest Remaining Time)
- 017번 답변에서는 SJF=비선점형 / SRT=선점형으로 정확히 구분했으므로, 018번 답변도 그 내용과 일치시켜 다시 정리할 것
- 정정 팁: 선점형 예시는 "Round Robin, SRT, (우선순위 선점형)" / 비선점형 예시는 "FCFS, SJF, Priority Scheduling(비선점형), HRN" 식으로 SJF와 SRT를 짝지어 기억하면 헷갈리지 않음

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변:

Time Quantum은 한 프로세스가 CPU를 사용하기 시작할 때, CPU를 사용할 수 있는 제한시간을 의미합니다.

Time Quantum이 길 경우, 프로세스의 CPU 할당을 교체하는 컨텍스트 스위칭이 적게 일어나기 때문에 그만큼의 지연 시간이 감소하지만, 하나의 프로세스가 CPU를 차지하는 시간이 늘어나기 때문에 각 프로세스의 지연 시간이 늘어난다는 단점이 있습니다.

반면에 Time Quantum이 너무 짧을 경우, 여러 프로세스가 짧은 간격으로 CPU를 할당받아 각 프로세스에 대한 지연 시간이 감소하지만, 컨텍스트 스위칭의 빈도수가 늘어나는 만큼 지연시간이 늘어난다는 단점이 있습니다.

참고 자료:

- 클 때/작을 때의 trade-off를 대기 시간과 컨텍스트 스위칭 오버헤드 관점에서 균형 있게 설명한 좋은 답변
- 추가하면 좋은 표현: "Time Quantum이 너무 크면 사실상 FCFS와 유사해진다"는 비유를 넣으면 극단적인 경우를 더 쉽게 설명할 수 있음

## [COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.

답변:

멀티태스킹이 가능한 이유는 여러 프로세스가 매우 짧은 시간동안 CPU 자원을 번갈아가면서 사용할 수 있기 때문입니다. 이 과정에서 사용되는 것이 CPU 스케줄링과 컨텍스트 스위칭인데, CPU 스케줄링을 통해 Ready Queue에서 다음 실행할 적절한 프로세스를 선택하고, 컨텍스트 스위칭을 통해 기존에 실행 중인 프로세스를 다시 준비 상태로 저장하고, 새로운 프로세스의 진행 상황을 복원하여 이를 실행하는 작업을 통해 여러 프로세스가 CPU 자원을 번갈아가면서 사용합니다. 따라서 실제로 여러 프로세스가 동시에 실행되는 것은 아니지만, 매우 짧은 시간마다 CPU를 번갈아가면서 사용하기 때문에 여러 프로그램이 동시에 실행되는 것처럼 보여 멀티태스킹이 가능합니다.

참고 자료:

- CPU 스케줄링과 컨텍스트 스위칭의 역할을 각각 짚고, 결론까지 자연스럽게 연결한 완성도 높은 답변
- 한 문장이 너무 길게 이어지는 편이라, 실제 면접에서는 두세 문장 단위로 끊어 말하는 연습을 해두면 더 안정적으로 전달됨 (예: "① 멀티태스킹의 원리 한 문장 → ② 스케줄링의 역할 → ③ 컨텍스트 스위칭의 역할 → ④ 결론")
