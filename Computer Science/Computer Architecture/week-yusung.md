## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변:

- ALU (Arithmetic Logic Unit, 산술 논리 장치)
  - CPU 내부에서 실제 연산을 수행하는 핵심 부품
  - 산술 연산: 덧셈, 뺄셈, 곱셈, 나눗셈 등
  - 논리 연산: AND, OR, NOT, XOR 등
  - 비교 연산 결과에 따라 Zero, Carry, Overflow 등의 플래그를 설정
- 제어장치 (Control Unit)
  - CPU 전체의 동작을 지휘, 통제하는 부품
  - 명령어를 해석(opcode 분석)하고 ALU, 레지스터, 메모리 등에 제어 신호를 전달
  - 명령어 사이클(Fetch → Decode → Execute)의 흐름을 조율, 2번 문항 참고
- 레지스터 (Register)
  - CPU 내부에 위치한 고속 임시 저장 공간
  - PC(Program Counter), IR(Instruction Register), 범용 레지스터 등이 존재
  - 메모리보다 용량은 작지만 접근 속도가 가장 빠름
  - ALU가 연산할 데이터와 결과를 일시적으로 보관

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변:

- 명령어 사이클(Instruction Cycle)
  - CPU가 메모리에서 명령어를 가져와 해석하고 실행하는 일련의 과정
  - 전원이 켜진 후 꺼질 때까지 Fetch → Decode → Execute를 반복
- Fetch (명령어 인출)
  - PC(Program Counter)가 가리키는 메모리 주소에서 명령어를 읽어 IR(Instruction Register)에 저장
  - Fetch 완료 후 PC는 다음 명령어 주소로 자동 증가
- Decode (명령어 해독)
  - IR에 저장된 명령어의 opcode, operand, 주소 모드 등을 해석
  - 제어장치가 ALU와 레지스터에 어떤 연산을 수행할지 제어 신호를 설정
- Execute (명령어 실행)
  - 해석된 명령에 따라 ALU 연산, 메모리 읽기/쓰기, 분기 등을 실제 수행
  - 연산 결과는 레지스터 또는 메모리에 저장

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변:

- 컴파일 (Compile)
  - 개발자가 작성한 고수준 소스 코드를 기계어(또는 어셈블리어)로 번역
  - 컴파일 결과물은 목적 파일(object file)로 생성
  - 문법 오류, 타입 오류 등을 이 단계에서 검출
- 링킹 (Link)
  - 여러 목적 파일과 외부 라이브러리를 하나의 실행 파일로 결합
  - 함수, 변수의 참조 주소를 실제 주소로 연결(심볼 해석)
  - 정적/동적 링크 방식에 따라 라이브러리 포함 여부가 달라짐
- 로딩 (Load)
  - 디스크에 저장된 실행 파일을 주기억장치(RAM)에 적재
  - 프로세스에 필요한 메모리 공간 할당, 재배치, 세그먼트/페이지 설정 수행
  - 로더(Loader) 또는 OS가 담당
- 실행 (Execute)
  - CPU가 메모리에 적재된 명령어를 순차적으로 Fetch → Decode → Execute
  - 프로그램 카운터(PC)를 기준으로 명령어를 하나씩 처리하며 프로그램 동작

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변:

- 컴파일 언어 (Compiled Language)
  - 실행 전에 소스 코드 전체를 기계어로 번역(컴파일)하여 실행 파일 생성
  - 대표 예: C, C++, Go, Rust
  - 장점: 실행 속도가 빠름, 배포 시 실행 파일만 전달하면 됨
  - 단점: 컴파일 시간 소요, 플랫폼별 빌드 필요, 런타임 에러 발견이 늦을 수 있음
- 인터프리터 언어 (Interpreted Language)
  - 소스 코드를 한 줄씩 읽어 즉시 해석·실행
  - 대표 예: Python, JavaScript, Ruby
  - 장점: 수정 후 바로 실행 가능, 디버깅·개발 편의성 높음
  - 단점: 매 실행마다 해석 필요 → 상대적으로 실행 속도 느림
- 차이
  - 번역 시점: 컴파일 언어는 실행 전, 인터프리터 언어는 실행 중
  - 결과물: 컴파일 언어는 독립 실행 파일, 인터프리터 언어는 인터프리터 위에서 동작

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변:

- JIT (Just-In-Time) 컴파일
  - 프로그램 실행 중(runtime)에 코드를 기계어로 번역하는 방식
  - 인터프리터처럼 실행하면서, 자주 실행되는 코드(hot spot)만 선별적으로 컴파일
  - Java(JVM), .NET, V8(JavaScript 엔진) 등에서 사용
- 장점
  - 순수 인터프리터 대비 실행 속도 향상
  - 실행 시점의 정보(분기 패턴, 타입 등)를 활용한 최적화 가능
  - 플랫폼 독립적인 바이트코드 + 네이티브 수준 성능의 절충
- 단점
  - 최초 실행 시 컴파일 오버헤드 발생 → cold start가 느릴 수 있음
  - 컴파일된 코드를 메모리에 캐싱 → 메모리 사용량 증가
  - AOT(Ahead-Of-Time) 컴파일 대비 최대 성능은 다소 낮을 수 있음

## [COMMON-006] 프로그램과 프로세스의 차이에 대해 설명해 주세요.

답변:

- 프로그램 (Program)
  - 디스크(HDD/SSD)에 저장된 실행 가능한 파일
  - 명령어와 데이터의 정적(static) 집합
  - 실행되기 전까지는 수동적인 존재
- 프로세스 (Process)
  - 프로그램이 메모리에 로드되어 CPU에서 실행 중인 상태
  - OS로부터 PID, 메모리, CPU 시간 등 자원을 할당받은 동적(dynamic) 실체
- 차이
  - 프로그램 1개 → 프로세스 N개 실행 가능 (예: Chrome 탭 여러 개)

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

답변:

- 프로세스 (Process)
  - OS가 자원을 할당하는 기본 단위
  - Code, Data, Heap, Stack 영역을 독립적으로 보유
  - 다른 프로세스의 메모리에 직접 접근 불가 (프로세스 간 격리)
- 스레드 (Thread)
  - 프로세스 내부에서 CPU를 할당받아 실행되는 최소 단위
  - 같은 프로세스 내 스레드들은 Code, Data, Heap 영역을 공유
  - 각 스레드는 독립적인 Stack과 PC(Program Counter)를 가짐
- 차이
  - 생성, 전환 비용: 스레드가 프로세스보다 가벼움 (Lightweight Process)
  - 통신: 스레드는 공유 메모리로 쉽게 통신, 프로세스는 IPC 필요
  - 안정성: 한 스레드 오류가 같은 프로세스의 다른 스레드에 영향 가능

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.

답변:

- PCB (Process Control Block, 프로세스 제어 블록)
  - OS가 각 프로세스를 관리하기 위해 유지하는 정보 구조체
  - 프로세스 생성 시 만들어지고, 종료 시 제거됨
- PCB에 포함되는 주요 정보
  - PID (Process ID), 프로세스 상태 (Ready, Running, Blocked 등)
  - CPU 레지스터 값 (PC, SP 등), CPU 스케줄링 정보 (우선순위, 큐 포인터)
  - 메모리 관리 정보 (페이지 테이블, 메모리 한계)
  - I/O 상태 정보 (할당된 장치, 열린 파일 목록)
- OS가 PCB를 사용하는 이유
  - 컨텍스트 스위칭 시 현재 실행 상태를 저장·복원하기 위해
  - CPU 스케줄링, 메모리 관리, I/O 관리 등 프로세스 단위 제어에 필수

## [COMMON-009] 컨텍스트 스위칭이 무엇이고, 언제 발생하는지 설명해 주세요.

답변:

- 컨텍스트 스위칭 (Context Switching)
  - CPU가 현재 실행 중인 프로세스(또는 스레드)의 상태를 PCB에 저장하고, 다른 프로세스의 상태를 복원하여 실행을 전환하는 과정
  - CPU는 한 번에 하나의 프로세스만 실행하지만, 빠른 전환으로 동시 실행처럼 보이게 함
- 발생 시점
  - Time Quantum(타임 슬라이스) 만료 → Round Robin 등 선점형 스케줄링
  - 더 높은 우선순위 프로세스가 Ready 상태가 될 때
  - Running 프로세스가 I/O 대기(Blocked) 상태로 전환될 때
  - I/O 완료 후 Blocked → Ready 상태로 복귀할 때
  - 인터럽트 발생 시
- 오버헤드
  - 레지스터 저장/복원, 캐시·TLB 플러시 등으로 순수 연산 시간이 아닌 추가 비용 발생

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요.

답변:

- 프로세스 컨텍스트 스위칭
  - 서로 다른 프로세스 간 CPU 실행 권한을 전환
  - PCB 전체(레지스터, 메모리 맵, 파일 디스크립터 등)를 저장, 복원
  - MMU(주소 변환) 변경, TLB 플러시, 캐시 무효화 등 추가 작업 필요
  - 오버헤드가 크고 비용이 높음
- 스레드 컨텍스트 스위칭
  - 같은 프로세스 내 스레드 간 CPU 실행 권한을 전환
  - Code/Data/Heap 메모리 공간은 공유 → MMU 변경 불필요
  - Stack, PC, 레지스터 등 스레드 고유 정보만 저장·복원
  - 프로세스 컨텍스트 스위칭보다 훨씬 빠르고 가벼움
- 핵심 차이
  - 메모리 공간 공유 여부가 오버헤드 크기를 결정
  - 멀티스레드가 멀티프로세스보다 전환 비용이 낮은 이유

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변:

- 유저 모드 (User Mode)
  - 일반 애플리케이션이 실행되는 모드
  - 제한된 권한으로 동작 → 하드웨어·OS 핵심 자원에 직접 접근 불가
  - 잘못된 코드로 인한 시스템 전체 장애를 방지
- 커널 모드 (Kernel Mode / Supervisor Mode)
  - OS 커널이 실행되는 모드
  - CPU, 메모리, I/O 장치 등 모든 하드웨어에 직접 접근 가능
  - 프로세스 관리, 메모리 관리, 파일 시스템, 장치 드라이버 등 핵심 기능 수행
- 구분하는 이유
  - 보안: 악의적, 버그 있는 프로그램이 시스템 자원을 함부로 조작하는 것을 차단
  - 안정성: 유저 프로그램 오류가 OS 전체에 영향을 주지 않도록 격리
  - 자원 보호: CPU의 특권 레벨(privilege level)을 통해 접근 권한을 엄격히 통제

## [COMMON-012] 시스템 콜이 무엇이고, 사용자 프로그램이 시스템 콜을 호출하는 이유를 설명해 주세요.

답변:

- 시스템 콜 (System Call)
  - 유저 모드에서 실행 중인 프로그램이 OS(커널)의 기능을 요청하기 위한 인터페이스
  - OS가 제공하는 API의 최하위 계층
- 사용자 프로그램이 시스템 콜을 호출하는 이유
  - 하드웨어 접근 제한: 유저 모드에서는 I/O, 메모리 관리 등 직접 수행 불가
  - 보안: OS가 권한 검사 후 안전하게 자원을 대리 처리
  - 추상화: 복잡한 하드웨어 조작을 OS에 위임하고 간단한 API로 사용
  - 자원 관리: 파일, 네트워크, 프로세스 등 OS가 관리하는 자원에 접근하기 위해 필수

## [COMMON-013] 시스템 콜이 실행될 때 유저 모드에서 커널 모드로 전환되는 과정을 설명해 주세요.

답변:

- 1. 시스템 콜 호출
  - 유저 프로그램이 라이브러리 함수를 호출
  - 라이브러리가 시스템 콜 번호와 인자를 레지스터에 설정
- 2. 트랩(Trap) 발생
  - `syscall`/`int 0x80`/`svc` 등 특수 명령어로 소프트웨어 인터럽트 발생
  - CPU가 유저 모드 → 커널 모드로 전환 (특권 레벨 변경)
- 3. 컨텍스트 저장
  - 현재 유저 프로세스의 PC, 레지스터, 스택 포인터 등을 커널 스택에 저장
- 4. 커널에서 처리
  - 시스템 콜 테이블에서 번호에 해당하는 커널 함수를 찾아 실행
  - 권한 검사 후 실제 I/O, 메모리, 프로세스 작업 수행
- 5. 유저 모드 복귀
  - 처리 결과를 레지스터에 저장
  - 저장해둔 유저 프로세스 컨텍스트 복원 → 유저 모드로 복귀 후 다음 명령어 실행

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.

답변:

- 인터럽트 (Interrupt)
  - CPU가 현재 작업을 일시 중단하고 우선 처리해야 할 이벤트에 응답하는 메커니즘
  - I/O 완료, 타이머 만료, 하드웨어 오류, 시스템 콜 등 다양한 원인으로 발생
- CPU 처리 과정
  - 현재 실행 중인 명령어 완료 후 인터럽트 신호 인식
  - 현재 PC, 레지스터 등 CPU 상태를 스택에 저장
  - 인터럽트 벡터 테이블(IVT)에서 해당 ISR(Interrupt Service Routine) 주소 조회
  - ISR 실행 → 인터럽트 원인 처리
  - 저장된 상태 복원 후 원래 작업 재개
- 운영체제 처리
  - 하드웨어 인터럽트: 커널 모드로 전환하여 ISR 실행, I/O 완료 시 대기 프로세스를 Ready로 변경
  - 타이머 인터럽트: CPU 스케줄링 트리거 (Time Quantum 관리)
  - 인터럽트 처리 후 필요 시 컨텍스트 스위칭 수행

## [COMMON-015] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이에 대해 설명해 주세요.

답변:

- 하드웨어 인터럽트 (Hardware Interrupt)
  - 외부 하드웨어 장치(키보드, 디스크, NIC, 타이머 등)가 발생시키는 전기적 신호
  - CPU와 비동기적으로 발생 → 예측하기 어려운 시점
  - IRQ(Interrupt Request) 라인을 통해 CPU에 전달
  - 예: 디스크 I/O 완료, 네트워크 패킷 수신, 타이머 만료
- 소프트웨어 인터럽트 (Software Interrupt / Trap)
  - 프로그램 실행 중 의도적으로 또는 예외적으로 발생
  - `int`, `syscall`, `trap` 명령어 또는 0으로 나누기, 페이지 폴트 등 예외로 발생
  - 동기적(synchronous) → 특정 코드 실행 시점에 발생
  - 예: 시스템 콜, 디버깅 브레이크포인트, 예외 처리
- 차이
  - 발생 주체: 하드웨어 장치 vs 소프트웨어(프로그램/CPU)
  - 발생 시점: 비동기 vs 동기
  - 목적: I/O·타이머 등 외부 이벤트 처리 vs OS 서비스 요청·예외 처리

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변:

- New (생성)
  - 프로세스가 생성되었으나 아직 OS에 의해 승인, 초기화되지 않은 상태
  - PCB 생성 및 메모리 할당 준비 단계
- Ready (준비)
  - 실행에 필요한 모든 자원을 할당받아 CPU만 기다리는 상태
  - Ready Queue에 대기 중
- Running (실행)
  - CPU를 할당받아 현재 명령어를 실행 중인 상태
  - CPU 코어 수만큼 동시에 Running 가능
- Blocked / Waiting (대기/블록)
  - I/O, 자식 프로세스 종료, 세maphore 등 이벤트를 기다리며 CPU를 사용하지 않는 상태
  - I/O 완료 등 이벤트 발생 시 Ready로 전환
- Terminated / Exit (종료)
  - 실행을 완료하거나 강제 종료된 상태
  - PCB 및 할당 자원 반납 후 제거

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변:

- CPU 스케줄러가 필요한 이유
  - CPU는 한 번에 하나(또는 코어당 하나)의 프로세스만 실행 가능
  - Ready Queue에 여러 프로세스가 대기 → 누구를 먼저 실행할지 결정 필요
  - CPU 이용률 극대화, 응답 시간·대기 시간 최소화, 공정성 확보
- 대표적인 스케줄링 알고리즘
  - FCFS (First Come First Served): 도착 순서대로 실행, 비선점형, 구현 간단
  - SJF (Shortest Job First): 예상 실행 시간이 짧은 프로세스 우선, 평균 대기 시간 최소화
  - Round Robin (RR): Time Quantum 단위로 순환 할당, 선점형, 시분할 시스템에 적합
  - Priority Scheduling: 우선순위가 높은 프로세스 먼저 실행, 기아(starvation) 문제 가능
  - Multilevel Queue: 프로세스 유형별로 큐를 분리하여 각각 다른 알고리즘 적용

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

답변:

- 비선점형 스케줄링 (Non-Preemptive)
  - CPU를 할당받은 프로세스가 자발적으로 CPU를 반납할 때까지 실행
  - Running → Ready 전환이 프로세스 자신의 행동(I/O 요청, 종료)에 의해서만 발생
  - 대표: FCFS, Non-Preemptive SJF
  - 장점: 컨텍스트 스위칭 적음, 오버헤드 낮음
  - 단점: 긴 프로세스가 CPU 독점 → 응답 시간 길어질 수 있음
- 선점형 스케줄링 (Preemptive)
  - OS(스케줄러)가 Running 프로세스의 CPU 사용을 강제로 중단 가능
  - Time Quantum 만료, 높은 우선순위 프로세스 도착 시 즉시 전환
  - 대표: Round Robin, Preemptive SJF, Priority (선점형)
  - 장점: 빠른 응답 시간, 공정한 CPU 분배, 멀티태스킹에 적합
  - 단점: 잦은 컨텍스트 스위칭으로 오버헤드 증가

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변:

- Time Quantum이 너무 클 때
  - 한 프로세스가 긴 시간 CPU를 독점 → 다른 프로세스 대기 시간 증가
  - Quantum → ∞ 에 가까워지면 FCFS와 동일하게 동작
  - 시분할(time-sharing)의 목적 상실, 응답성 저하
  - 장점: 컨텍스트 스위칭 횟수 감소 → 오버헤드 감소
- Time Quantum이 너무 작을 때
  - 프로세스 전환이 매우 빈번 → 컨텍스트 스위칭 오버헤드 급증
  - 실제 연산보다 전환 비용 비중이 커져 CPU 처리량(throughput) 저하
  - 장점: 응답 시간(response time)은 매우 짧아짐
- 적절한 Quantum 선택
  - 일반적으로 수 ms ~ 수십 ms 수준 (컨텍스트 스위칭 비용 < Quantum)
  - 시스템 특성(대화형 vs 배치)에 따라 트레이드오프 조절

## [COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.

답변:

- 멀티태스킹 (Multitasking)
  - 단일 CPU(또는 코어)에서도 여러 프로그램이 동시에 실행되는 것처럼 보이는 기술
  - 실제로는 CPU 시간을 매우 짧은 단위로 나누어 번갈아 실행
- CPU 스케줄링의 역할
  - Ready Queue에서 다음 실행할 프로세스를 선택
  - Time Quantum, 우선순위 등 정책에 따라 CPU 시간을 공정·효율적으로 분배
  - I/O 대기 프로세스는 Blocked로 보내고, 다른 Ready 프로세스에 CPU 할당
  - 타이머 인터럽트를 통해 주기적으로 스케줄링 트리거
- 컨텍스트 스위칭의 역할
  - 프로세스 전환 시 현재 레지스터·PC 등 실행 상태를 PCB에 저장
  - 다음 프로세스의 저장된 상태를 복원하여 실행 재개
  - 빠른 전환(ms 이하)으로 사용자는 동시 실행으로 인식
- 결론
  - 스케줄링(누구를) + 컨텍스트 스위칭(어떻게 전환)이 결합되어 멀티태스킹 실현
