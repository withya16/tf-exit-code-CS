## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변: 명령어 사이클은 CPU가 하나의 명령어를 가져와 해석하고, 실행하는 반복 과정입니다. Fetch 단계에서는 프로그램 카운터가 가리키는 메모리 주소에서 명령어를 가져오고, Decode 단계에서는 가져온 명령어가 어떤 작업을 의미하는지 해석합니다. 마지막으로 Execute 단계에서 해석한 명령어를 실제로 실행합니다. 그래서 CPU는 프로그램을 한 번에 전체 실행하기보다, 명령어를 하나씩 가져와서 해석하고 실행하는 과정을 매우 빠르게 반복합니다.

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변: 컴파일에서 소스 코드가 기계어 또는 중간 코드로 변환되고, 링킹에서 오브젝트 파일과 라이브러리를 연결하여 실행 파일을 만듭니다. 이후 로딩에서 운영체제가 실행 파일을 메모리에 올리고, 프로세스 주소 공간을 구성하게 되며, CPU가 해당 프로세스의 명령어를 가져와 실행하면서 프로그램이 작동합니다.

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변: 컴파일 언어는 우선 실행 전에 소스 코드를 기계어나 실행 가능한 형태로 변환한 뒤 실행됩니다. 그래서 실행 속도가 빠른 편이지만, 플랫폼에 따라 별도의 컴파일이 필요할 수 있습니다.
Python과 같은 인터프리터 언어는 실행 시점에 코드를 한 줄씩 해석하며 실행하는 방식입니다. 그래서 실행 환경만 있으면 비교적 유연하게 실행할 수 있지만, 매번 해석 과정이 필요해서 일반적으로 컴파일 언어보다 느릴 수 있습니다.

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변: JIT는 Just-In-Time Complier의 약자로, 프로그램 실행 중에 자주 실행되는 코드를 기계어로 컴파일해 성능을 높이는 방식입니다. 자바의 경우, `.class` 바이트코드를 JVM이 실행하는데, 처음에는 인터프리터 방식으로 실행하다가 자주 호출되는 메서드나 반복되는 코드를 JIT 컴파일러가 네이티브 코드로 변환합니다. 이를 통해 반복 실행되는 코드의 성능을 높일 수 있습니다.
장점은 실행 중 실제 사용 패턴을 보고 최적화할 수 있고, 단점은 초기 실행 시점에는 컴파일 비용이 발생하고, JVM 같은 런타임 환경이 필요하다는 점입니다.

## [COMMON-006] 프로그램과 프로세스의 차이에 대해 설명해 주세요.

답변: 프로그램은 디스크에 저장되어 있는 정적인 실행 파일입니다. 예를 들어 `.exe` 파일이나 Java의 jar 파일처럼 아직 실행되지 않은 코드와 데이터의 집합입니다.
프로세스는 프로그램이 메모리에 올라와 실행 중인 상태입니다. 운영체제는 실행 중인 프로세스마다 독립적인 주소 공간, PCB, 스케줄링 정보 등을 관리합니다.

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

답변: 프로세스는 운영체제로부터 독립적인 주소 공간과 자원을 할당받아 실행되는 작업 단위입니다. 서로 다른 프로세스는 기본적으로 메모리를 공유하지 않아서, 안정성은 높지만, 통신을 하려면 IPC 같은 별도 방법이 필요합니다.
반면, 스레드는 하나의 프로세스 안에서 실행되는 흐름 단위로서 같은 프로세스의 스레드들은 Code, data, Heap 영역을 공유하고, 각자 별도의 Stack을 가집니다. 그래서 자원을 공유하기에 생성과 전환 비용이 비교적 작지만, 동시에 같은 데이터 접근하면 동시성 문제가 발생할 수 있습니다.

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.

답변: PCB는 Process Control Block의 약자로, 운영체제가 프로세스를 관리하기 위해 사용하는 자료구조입니다. PCB에는 프로세스 ID, 프로세스 상태, 프로그램 카운터, CPU 레지스터 값, 메모리 관리 정보, 스케줄링 정보 등이 저장됩니다.
이 PCB를 통해 운영체제는 중단된 프로세스의 필요한 상태 정보를 저장하고, 나중에 다시 실행할 수 있습니다.

## [COMMON-009] 컨텍스트 스위칭이 무엇이고, 언제 발생하는지 설명해 주세요.

답변: 컨텍스트 스위칭은 CPU가 실행 중이던 프로세스나 스레드를 멈추고, 다른 프로세스나 스레드를 실행하기 위해 실행 상태를 저장하고 복원하는 과정입니다.
타임 슬라이스나, I/O 대기가 발생했을 때, 더 높은 우선순위 작업이 들어왔을 때나 인터럽트가 발생한 경우 일어날 수 있습니다.
다만 저장과 복원 비용이 발생해서 잦은 발생 시에는 성능 저하가 일어날 수 있습니다.

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요.

답변: 프로세스 컨텍스트 스위칭은 서로 다른 프로세스 간 CPU 실행 대상을 바꾸는 것입니다. 프로세스마다 독립적인 주소 공간을 가지기 때문에 메모리 주소 공간 관련 정보도 바뀌어야 하고, TLB 같은 캐시 정보가 무효화 될 수 있어 비용이 큽니다.
반면, 스레드 컨텍스트 스위칭은 같은 프로세스 안의 스레드 간 실행 대상을 바꾸는 작업이고, Code, Data, Heap 영역을 공유하니까 주소 공간 전환 비용이 상대적으로 적습니다. 주로 각 스레드의 레지스터 값, 프로그램 카운터, 스택 정보를 바꾸는 것으로 알고 있습니다.

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변: 시스템의 안정성과 보안을 지키기 위해서라고 알고 있습니다. 유저는 일반 애플리케이션이 실행되는 모드이고, 하드웨어나 중요한 커널 자원에는 직접 접근할 수 없는 반면, 커널은 운영체제 커널이 실행되기에 메모리, 파일 시스템, 네트워크, 장치 같은 핵심 자원에 접근할 수 있습니다.
그래서 일반 프로그램은 시스템 콜을 통해 커널에게 요청하고, 커널이 검증 후 대신 작업을 수행하면서 버그나 악성 코드를 방지할 수 있습니다.

## [COMMON-012] 시스템 콜이 무엇이고, 사용자 프로그램이 시스템 콜을 호출하는 이유를 설명해 주세요.

답변: 시스템 콜은 사용자 프로그램이 운영체제 커널의 기능을 사용하기 위해 호출하는 인터페이스입니다. 일반 애플리케이션은 파일 입출력, 네트워크 통신, 프로세스 생성, 메모리 할당 같은 작업을 직접 수행할 수 없어서, 시스템 콜을 통해 커널 모드로 전환한 뒤 OS가 대신 작업을 수행합니다.

## [COMMON-013] 시스템 콜이 실행될 때 유저 모드에서 커널 모드로 전환되는 과정을 설명해 주세요.

답변: 사용자 프로그램은 시스템 콜 번호와 필요한 인자를 레지스터나 스택 등에 준비하고, 소프트웨어 인터럽트나 특수 명령어를 통해 CPU 모드를 유저 모드에서 커널 모드로 전환합니다.
여기서 커널은 시스템 콜 번호를 보고 어떤 작업인지 확인한 뒤, 커널 내부의 시스템 콜 핸들러를 실행하고, 작업이 끝나면 사용자 프로그램에 결과를 반환한 후, 다시 유저 모드로 돌아옵니다.

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.

답변: 인터럽트는 CPU가 현재 실행 중인 작업을 멈추고, 우선 처리해야 하는 이벤트를 처리하도록 하는 신호입니다. 인터럽트 벡터를 통해 해당 인터럽트를 처리할 핸들러로 이동해서 필요한 작업을 수행하고, 다시 이전 작업을 이어서 실행할 수 있습니다.

## [COMMON-015] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이에 대해 설명해 주세요.

답변: 하드웨어 인터럽트는 외부 하드웨어 장치가 CPU에게 이벤트 발생을 알리는 신호입니다. ( ex) 키보드 입력, 마우스 입력, 네트워크 패킷 수신, 디스크 I/O 완료) 그리고 소프트웨어 인터럽트는 프로그램 실행 중에 명령어에 의해 발생하는 인터럽트입니다. (ex) 시스템 콜)
둘 다 CPU의 현재 흐름을 잠시 중단하고 특정 핸들러로 실행하지만, 하드웨어는 외부, 소프트웨어는 내부 실행 흐름에서 발생한다는 차이가 있습니다.

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변: 프로세스의 상태에는 New, Ready, Running, Waiting, Terminated가 있습니다. 

- New는 프로세스가 생성 중인 상태
- Ready는 cpu만 할당받으면 실행할 수 있는 상태
- Running은 할당 받아서 실행 중인 상태
- Waiting 또는 Blocked는 I/O 작업이나 이벤트 완료를 기다리는 상태
- Terminated는 실행이 끝난 상태로 알고 있습니다.

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변: CPU 스케줄러는 여러 프로세스나 스레드 중에 어떤 작업에 CPU를 할당할지 결정하는 OS 구성 요소입니다. 이 CPU를 여러 프로세스나 스레드가 효율적으로 사용하기 위해 스케줄러가 필요하고, 대표적으로 FCFS(First Come First Served), SJF(Shortest Job First), Round Robin 등이 있습니다.
이 스케줄링 알고리즘은 응답 시간이나 처리량, 공정성, 대기 시간 등을 고려해서 선택됩니다.

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

답변: 선점형 스케줄링은 실행 중인 프로세스라도 운영체제가 CPU를 강제로 회수해서 다른 프로세스에 할당할 수 있는 방식입니다. 예를 들어 Round Robin처럼 타임 퀀텀이 끝나면 현재 프로세스를 멈추고, 다른 프로세스를 실행할 수 있습니다.
반면, 비선점형은 한 프로세스가 CPU를 할당받으면 작업이 끝나거나 자발적으로 대기 상태가 될 때까지 CPU를 계속 사용합니다.

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변: 타임 퀀텀이 너무 작으면 프로세스가 자주 바껴서 컨텍스트 스위칭이 자주 발생하고, 실제 작업보다 전환 비용이 커져서 성능이 떨어질 수 있습니다.
반면에 타임 퀀텀이 너무 크면 한 프로세스가 CPU를 오래 점유하게 되니까 응답시간이 길어집니다. 그래서 타임 퀀텀은 컨텍스트 스위칭 비용과 응답성 사이의 균형을 고려해서 정해야 합니다.

## [COMMON-021] 파이프라이닝이 무엇이고, CPU 성능 향상에 어떻게 기여하는지 설명해 주세요.

답변: 파이프라이닝은 하나의 명령어 실행 과정을 여러 단계로 나누고, 여러 명령어가 각 단계를 동시에 지나가도록 하는 CPU 성능 향상 기법입니다.

이렇게 하면 하나의 명령어를 처리하는 시간 자체가 줄어드는 것은 아니지만, 일정 시간 동안 처리할 수 있는 명령어 수가 늘어나 전체 처리량이 증가합니다.

## [COMMON-023] CISC와 RISC의 차이에 대해 설명해 주세요.

답변: CISC는 Complex Instruction Set Computer의 약자로, 복잡하고 다양한 명령어 집합을 가진 구조입니다.
하나의 명령어가 여러 작업을 수행할 수 있어 코드 길이를 줄일 수 있지만, 명령어 해석과 실행이 복잡해질 수 있습니다. RISC는 Reduced Instruction Set Computer의 약자로, 단순하고 고정된 길이의 명령어를 사용하는 구조입니다. 명령어가 단순하고, 파이프라이닝과 병렬 처리에 유리합니다.
대표적으로 x86은 CISC 계열, ARM과 RISC-V 아키텍처는 RISC 계열로 분류하는 걸로 알고 있습니다.

## [COMMON-024] x86과 ARM 아키텍처의 차이에 대해 설명해 주세요.

## [COMMON-037] DMA가 무엇이고, 왜 필요한지 설명해 주세요.

## [COMMON-038] 운영체제가 하드웨어 자원을 추상화한다는 말의 의미를 설명해 주세요.

답변: 운영체제가 하드웨어 자원을 추상화한다는 것은, 사용자가 복잡한 하드웨어 세부사항을 직접 다루지 않아도 일관된 인터페이스로 자원을 사용하게 해준다는 의미로 이해하고 있습니다.
예를 들어서, 개발자가 디스크의 물리적인 위치나 장치 제어 방식을 몰라도 파일이라는 추상화된 개념을 통해 데이터를 읽고 쓸 수 있고, 네트워크도 소켓이라는 인터페이스를 통해 사용하거나, 메모리도 가상 메모리 덕에 각 프로세스가 독립적인 주소 공간을 가진 것처럼 사용할 수 있습니다.
즉 운영체제는 CPU, 메모리, 디스크, 네트워크 같은 하드웨어 자원을 프로세스, 파일, 소켓, 가상 메모리 같은 개념으로 추상화해서 개발자가 쉽게 사용하도록 하는 구조로 알고 있습니다.

## [COMMON-039] 부팅 과정에서 운영체제가 메모리에 올라오는 흐름을 간단히 설명해 주세요.

답변: 

- 컴퓨터가 켜지면 먼저 BIOS나 UEFI가 실행되어 하드웨어를 초기화하고 부팅 가능한 장치를 찾습니다.
- 그 다음 부트로더가 실행되고, 여기서 부트로더는 디스크에 저장된 운영체제 커널을 메모리에 적재하고 실행을 넘깁니다.
- 그래서 커널이 실행되면 메모리, 프로세스 관리, 파일 시스템, 장치 드라이버 같은 우영체제 핵심 기능을 초기화합니다.
- 이후에 이제 init이나 systemd 같은 첫 번째 사용자 공간 프로세스가 실행되고, 필요한 서비스들이 차례로 시작됩니다.
- 리눅스 서버에서 부팅 후에 Docker, Nginx, DB 같은 서비스가 자동으로 올라오는 것도 이런 초기화 과정 이후에 systemd 등이 서비스를 실행하는 흐름으로 진행됩니다.
