## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변:
ALU는 Arithmetic Logic Unit의 약자로, CPU 내에서 덧셈, 뺄셈등의 산술 연산과 And, Or과 같은 논리 연산을 수행합니다.
CPU 내의 Control Unit은 전체 시스템을 통제하고 지시하며 적절한 순서로 명령을 꺼내고 해석하여 필요한 장치에 제어 신호를 보냅니다.
레지스터는 CPU 내에서 명령이나 연산 결과를 일시적으로 저장하는 임시기억장치입니다. PC내에 존재하는 기억 장치 중 가장 속도가 빠르며 크기가 작습니다.

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변:
명령어 사이클은 CPU가 메모리에서 하나의 명령어를 가져와 해석하고 실행하는 단계이며, 전원이 켜진 이후 꺼질때까지 무한 반복하는 행위입니다.
Fetch 단계에서는 프로그램 카운터가 가리키는 위치로부터 명령어를 가져옵니다.
Indirect 단계에서는 명령어에 포함된 주소 정보를 이용해 실제 저장된 메모리 주소를 확보합니다.
Excute 단계에서는 가져온 명령어를 실제로 수행하는 단계입니다. (보통 국내 자료에서 Decode는 단계로 분류하지 않고 Excute 내부에 포함시키는 듯)

참고 자료:
https://en.wikipedia.org/wiki/Instruction_cycle
https://devrabbit22.tistory.com/91

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변:
개발자가 프로그램을 하이레벨의 코드로 작성한 내용을 컴파일 과정을 통해 기계가 인식할 수 있는 로우레벨 언어(기계어)로 변환하게 됩니다. 이후 링킹을 통해 여러개의 목적파일과 필요한 라이브러리 파일을 하나로 묶어 실행 파일을 만듭니다.
로딩은 완성된 실행 파일을 보조 기억 장치에서 주 기억 장치로 적재하여 프로그램이 실행될 수 있도록 준비하는 과정이며, 이후 CPU가 적재된 명령어를 순차적으로 처리하여 실행합니다.

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변:
JIT는 Just-In-Time의 약자이며 프로그램을 실행하기 전 전부 기계어로 번역하는 대신, 런타임에 필요한 코드만 기계어로 번역하여 실행속도를 높일 수 있는 방식입니다. Java에서 주로 사용되며 동적 컴파일이라고도 불리기도 합니다.
인터프리터의 느린 실행 속도를 개선할 수는 있지만, 최초 실행시에는 느린 단점이 있습니다.

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

동일 프로세스 내의 스레드들은 메모리의 Code, Data, Heap 영역은 공유하지만 각자 독립적인 Stack 영역을 가집니다.

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.

답변:
Process Context BLock의 약자이며, 각 프로세스는 고유한 PCB를 가지고 운영체제는 PCB를 통해 CPU 주기가 끝나도 다음 주기에서 (컨텍스트 스위칭이 일어나도) 작업하던 내용을 불러올 수 있습니다.

## [COMMON-009] 컨텍스트 스위칭이 무엇이고, 언제 발생하는지 설명해 주세요.

답변:
컨텍스트 스위칭은 현재 실행 중인 프로세스의 상태를 저장 후 중단하고, 다른 프로세스를 실행할 수 있도록 하는 과정입니다.
현재 프로세스의 이번 주기 내 CPU 시간을 모두 사용했을 때에 발생하게 됩니다.

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요.

답변:
스레드 컨텍스트 스위칭은 실행중인 프로세스의 context 백업만 이루어지면 되지만,
프로세스 컨텍스트 스위칭은 추가로 캐시 플러싱을 진행하고, TLB를 비우고 MMU를 변경해야 합니다.
때문에 스레드 컨텍스트 스위칭이 더 빠르게 동작하게 됩니다. (오버헤드가 작습니다.)

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변:
운영체제에서 유저 모드와 커널 모드를 구분하는 이유는, 운영체제와 시스템 자원에 대한 접근 권한을 구분하려는 보안적인 이유입니다.
커널 모드는 운영 체제의 핵심 기능을 수행하고 하드웨어와 직접 상호작용할 수 있는 권한이 있습니다.
유저 모드는 제한된 권한을 가지며, 자원에 직접 접근할 수 없습니다. 시스템 콜을 사용하여 커널 모드로의 진입이 가능합니다.

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.

답변:
인터럽트는 프로세스가 커널에게 어떠한 이벤트가 일어났다고 알려주는 방법입니다.
인터럽트가 발생하면 CPU는 현재 실행중이던 작업 상태를 메모리에 저장하고, Interrupt Service Routine을 실행하여 해당 인터럽트를 처리합니다.
처리가 끝나면 이전에 저장해둔 작업 상태를 불러와 원래 작업을 계속합니다.

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변: Five-State Process Model을 기준으로 설명합니다.. 이거로 배워가지고 다른건 몰랐네요

New - 프로세스가 막 만들어진 상태이며 프로세스 실행에 필요한 PCB 생성 및 메모리 할당 등을 처리하는 단계
Ready - New에서의 처리가 끝나고, 실행을 하기 위해 기다리는 단계. CPU를 할당해주면 바로 실행 가능한 단계
Running - 프로세스가 CPU를 점유하고 실행하고 있는 상태
Blocked - I/O 요청과 같이 CPU를 사용하지 않고 기다리고 있는 상태. (스케줄링 제외)
Exit - 프로세스가 종료될 때에 할당 받았던 자원을 반납하는 단계

참고 자료:
https://velog.io/@zooneon/OS-Process-State

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변:
Time Quantum이 너무 크게 잡히면 하나의 프로세스가 주기를 마치고 다음 스케줄링까지 기다리게 되는 시간이 너무 길어집니다.
반대로 너무 작을시에는 컨텍스트 스위칭이 자주 일어나게 되어 처리 효율이 떨어지게 됩니다.

## [COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.

답변:
단일 코어에서도, 혹은 코어의 개수보다 많은 프로세스를 운용할 수 있는 것은 CPU의 사용 시간을 분배하는 스케줄링 덕분입니다.
스케줄링 정책에 따라 프로세스마다 돌아가며 CPU를 분배하며, 이때 프로세스 전환이 될 경우에는 컨텍스트 스위칭을 통해 PCB에 현재 상황을 저장합니다.
