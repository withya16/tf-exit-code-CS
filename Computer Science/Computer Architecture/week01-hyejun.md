## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변: ALU는 산술연산(덧셈, 뺄셈 등)과 논리연산 수행(AND, OR, 비교) 등을 수행한다. 제어장치는 명령어를 해석하고 CPU 내부 구성 요소에 제어신호를 전달한다. 레지스터는 명령어, 주소, 연산결과 등을 임시로 저장하는 매우 빠른 기억장치이다.

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변: 명령어 사이클은 cpu가 하나의 명령어를 처리하는 반복 과정이다. FETCH 단계에서는 PC가 가리키는 주소에서 명령어를 가져오고, DECODE 단계에서는 명령어의 종류와 피연산자를 해석한다. EXCUTE단계에서는 ALU 연산, 메모리 접근, 분기 등 실제 작업을 수행 -> 결과를 저장한다.

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변: 컴파일러는 소스코드 -> 목적파일(기계어 포함) 로 변환하는 역할을 담당한다. 링커는 여러 목적 파일, 라이브러리를 결합하고 symbol주소를 해결하여 실행파일을 만든다. 로더는 실행파일을 메모리에 배치하고 실행환경을 구성 -> 이후 CPU가 프로그램의 시작지점부터 명령어를 수행한다.

참고 자료:

---

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변: 컴파일 방식은 프로그램 실행 전에 소스코드 전체를 기계어 or 중간코드로 변환. 인터프리터 방식은 실행 중 소스코드나 중간코드를 순차 해석. 컴파일방식이 실행속도가 빠르고, 인터프리터 방식은 실행과 디버깅이 유연. 두 방식이 함께 사용되기도 한다.

참고 자료:

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변: JIT 컴파일은 프로그램 실행 중 자주 실행되는 코드를 기계어로 변환하는 방식. 실제 실행 정보를 이용해 최적화가 가능하여 반복 실행 성능이 좋아진다. 반면 초기 컴파일 시간과 추가메모리가 필요하며, 실행 중 컴파일 때문에 일시적 지연 발생 가능.

참고 자료:

## [COMMON-006] 프로그램과 프로세스의 차이에 대해 설명해 주세요.

답변: 프로그램은 저장장치에 존재하는 실행 가능한 코드와 데이터, 실행되지 않은 정적 상태. 프로세스는 프로그램이 실행되면서 메모리, cpu상태, 파일 등 자원을 할당받은 동적 실행 단위. 하나의 프로그램으로 여러 프로세스 실행 가능

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

답변: 프로세스는 독립적 주소공간, 자원을 가진 실행 단위. 같은 프로세스의 스레드들은 코드, 데이터, 힙 등 자원을 공유하지만 각각 별도의 스택, 레지스터, 프로그램 카운터를 가진다. 스레드는 생성과 전환 비용이 작지만 공유 자원에 대한 동기화가 필요하다.

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.

답변: PCB는 운영체제가 프로세스를 관리하기 위해 유지하는 자료구조. 프로세스 ID, 상태, PC, 레지스터, 스케줄링 정보, 메모리정보, 열린 파일 등의 정보 저장. 운영체제는 PCB를 이용해서 중단된 프로세스의 상태를 보관하고 나중에 재실행.

참고 자료:

## [COMMON-009] 컨텍스트 스위칭이 무엇이고, 언제 발생하는지 설명해 주세요.

답변: CPU가 현재 실행 중인 작업의 상태를 저장하고 다른 작업의 상태를 복원하는 과정. 타임 퀀텀이 끝나거나, 현재 작업이 i/o를 기다리거나, 더 높은 우선순위의 작업이 준비 상태가 되면 발생. 전환 과정 자체는 실제 작업을 수행하지 않으므로 오버헤드 발생

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요.

답변: 프로세스 간 전환에서는 레지스터뿐만 아니라 주소 공간, 메모리 관리 정보도 변경해야 한다. 같은 프로세스 안의 스레드 간 전환은 주소공간 공유 -> 주로 레지스터, 스택 등 스레드 상태를 교체. 따라서 보통 스레드 컨텍스트 비용이 더 적다.

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변: 유저모드에서는 응용프로그램이 제한된 권한으로 실행, 커널모드에서는 운영체제가 하드웨어와 전체 메모리에 접근 가능. 이를 구분하면 사용자 프로그램의 오류 or 악성 동작이 운영체제와 다른 프로세스에 직접 영향을 주는 것을 방지

## [COMMON-012] 시스템 콜이 무엇이고, 사용자 프로그램이 시스템 콜을 호출하는 이유를 설명해 주세요.

답변: 시스템 콜은 사용자 프로그램이 운영체제 커널 기능을 요청하는 공식적인 인터페이스이다. 파일 입출력, 프로세스 생성, 메모리 할당, 네트워크 통신 등은 보호된 자원에 접근해야 하기 때문에 시스템 콜을 통해 커널에 요청해야 한다.

## [COMMON-013] 시스템 콜이 실행될 때 유저 모드에서 커널 모드로 전환되는 과정을 설명해 주세요.

답변: 사용자 프로그램은 시스템 콜 번호, 인자를 설정한 후 전용 시스템 콜 명령어를 실행. CPU는 현재 실행상태를 저장하고 권한 수준을 커널모드로 변경한 뒤, 등록된 커널 처리 루틴으로 이동.

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.

답변: 인터럽트는 장치나 시스템이 cpu에 특정 사건의 처리를 요청하는 신호. CPU는 현재 실행상태를 저장하고 인터럽트 벡터를 통해 해당 인터럽트 서비스 루틴을 실행한다. 운체가 원인을 처리한 뒤 저장된 상태를 복원하면 기존 작업이 이어서 실행된다.

## [COMMON-015] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이에 대해 설명해 주세요.

답변: 하드웨어 인터럽트는 키보드, 네트워크 장치, 타이머 등 CPU 외부장치가 발생시키며, 현재 명령어와 비동기적으로 발생. 소프트웨어 인터럽트는 프로그램이 특정 명령어를 실행하거나 오류, 예외가 발생했을 때 CPU 내부에서 발생

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변: 실행 가능한 프로세스, 스레드 수가 cpu수보다 많기 때문에 스케줄러가 다음 실행 대상을 선택해야 한다. 대표적인 알고리즘으로 FCFS, SJF, 우선순위 스케줄링, RR 가 있다. 응답시간, 처리량, 공정성 사이에서 서로 다른 특성을 가진다.

참고 자료:

---

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

답변: 선점형 스케줄링 - 운영체제가 실행중인 작업에서 CPU를 강제로 회수하여 다른 작업에 할당 가능. 비선점형 스케줄링 - 작업이 종료되거나 대기상태로 이동하거나 스스로 CPU를 반납할 때까지 계속 실행. 선점형은 응답성이 좋지만 컨텍스트 스위칭, 동기화 부담이 크다.

참고 자료:

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변: Time Quantum이 너무 크면 컨텍스트 스위칭 횟수는 감소 but FCFS와 비슷하게 동작하여 짧은 작업의 응답시간이 길어질 수 있다. 너무 작으면 응답성, 공정성은 좋아지지만 컨텍스트 스위칭이 지나치게 자주 발생하여 CPU 오버헤드가 증가.

참고 자료:

## [COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.

답변: CPU 스케줄러가 여러 작업에 짧은 시간 단위로 CPU를 번갈아 할당하고, 컨텍스트 스위칭을 통해 각 작업의 실행 상태를 저장하고 복원하기 때문에 멀티태스킹 가능. 전환 속도가 매우 빠르기 때문에 사용자는 여러 프로그램이 동시에 실행되는 것처럼 느낀다. 멀티코어 환경에서는 일부 작업이 실제로 병렬 실행될수도 있다.
