## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변:

* CPU는 크게 ALU, 제어장치, 레지스터로 구성된다. 
* ALU는 덧셈과 뺄셈 같은 산술 연산과 AND, OR, 비교 같은 논리 연산을 수행한다. 
* 제어장치는 명령어를 해석하고 CPU 내부의 각 장치에 제어 신호를 보내 명령어가 올바른 순서로 실행되도록 한다. 
* 레지스터는 CPU 내부에 있는 매우 빠른 저장 공간으로, 연산할 데이터와 명령어 주소, 중간 연산 결과 등을 임시로 저장한다.

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변:

명령어 사이클은 CPU가 하나의 명령어를 가져와 해석하고 실행하는 과정이다. 

* Fetch 단계에서는 프로그램 카운터가 가리키는 메모리 주소에서 다음 명령어를 가져온다. 
* Decode 단계에서는 명령어의 연산 종류와 필요한 피연산자를 해석한다. 
* Execute 단계에서는 ALU 연산, 메모리 접근, 분기 처리 등 실제 작업을 수행하고 결과를 레지스터나 메모리에 저장한다. 

이 과정은 프로그램이 실행되는 동안 반복된다.

참고 자료:

https://www.geeksforgeeks.org/computer-organization-architecture/introduction-of-control-unit-and-its-design/

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변:

소스 코드는 컴파일 과정을 거쳐 기계어가 포함된 목적 파일로 변환된다. 링커는 여러 목적 파일과 라이브러리를 결합하고 서로 참조하는 함수와 변수의 주소를 연결하여 실행 파일을 만든다. 프로그램을 실행하면 로더가 실행 파일의 코드와 데이터를 메모리에 배치하고 필요한 공유 라이브러리를 연결한다. 이후 운영체제가 프로세스를 생성하고 프로그램의 시작 지점부터 명령어를 실행한다.

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변:

* 컴파일 방식은 프로그램을 실행하기 전에 소스 코드 전체를 기계어나 중간 코드로 변환한다. 변환 과정이 먼저 필요하지만 실행 시점에는 이미 번역된 코드를 사용하므로 일반적으로 실행 속도가 빠르다. 
* 인터프리터 방식은 프로그램 실행 중에 소스 코드나 중간 코드를 해석하면서 실행한다. 실행 전 별도의 전체 컴파일 과정이 필요하지 않지만 반복적인 해석 비용이 발생할 수 있다. 다만 실제 언어 구현에서는 컴파일과 인터프리터 방식을 함께 사용하기도 한다.

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변:

JIT 컴파일은 프로그램 실행 중에 자주 실행되는 코드나 성능상 중요한 코드를 기계어로 컴파일하는 방식이다. 실제 실행 정보를 바탕으로 자주 호출되는 함수나 타입 정보를 최적화할 수 있어 단순 인터프리터 방식보다 높은 성능을 낼 수 있다. 반면 실행 도중 컴파일이 수행되므로 초기 실행 시간이 길어질 수 있고, 컴파일 과정에서 CPU와 메모리를 추가로 사용한다. 실행 초기와 최적화 이후의 성능 차이가 발생할 수 있다는 점도 단점이다.

참고 자료:

https://v8.dev/blog/maglev

## [COMMON-006] 프로그램과 프로세스의 차이에 대해 설명해 주세요.

답변:

* 프로그램은 디스크에 저장된 명령어와 데이터의 집합으로, 아직 실행되지 않은 정적인 파일이다. 
* 프로세스는 프로그램이 실행되면서 메모리에 올라온 동적인 실행 단위이다. 하나의 프로그램을 여러 번 실행하면 동일한 프로그램을 기반으로 여러 개의 프로세스가 생성될 수 있다. 각 프로세스는 독립적인 주소 공간, 실행 상태, 레지스터 값, 프로세스 ID 등의 정보를 가진다.

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

답변:

프로세스는 독립적인 주소 공간과 운영체제 자원을 가지는 실행 단위이고, 스레드는 프로세스 내부에서 실제 명령어를 실행하는 단위이다. 

* 같은 프로세스에 속한 스레드들은 코드, 전역 데이터, 힙, 열린 파일 등을 공유한다. 
* 각 스레드는 독립적인 프로그램 카운터, 레지스터, 스택을 가진다. 스레드는 자원을 공유하므로 통신 비용이 작지만, 공유 데이터에 동시에 접근할 경우 동기화 문제가 발생할 수 있다.

참고 자료:

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.

답변:

PCB는 Process Control Block의 약자로, 운영체제가 각 프로세스를 관리하기 위해 유지하는 자료구조이다. PCB에는 프로세스 ID, 현재 상태, 프로그램 카운터, CPU 레지스터 값, 스케줄링 정보, 메모리 관리 정보, 열린 파일 정보 등이 저장된다. 운영체제는 PCB를 이용해 실행 중인 프로세스를 중단했다가 나중에 같은 지점부터 다시 실행할 수 있다. 프로세스 생성, 종료, 스케줄링, 자원 관리에도 PCB 정보가 사용된다.

참고 자료:

https://www.geeksforgeeks.org/operating-systems/process-control-block-in-os/

## [COMMON-009] 컨텍스트 스위칭이 무엇이고, 언제 발생하는지 설명해 주세요.

답변:

컨텍스트 스위칭은 CPU가 현재 실행 중인 프로세스나 스레드의 상태를 저장하고, 다른 프로세스나 스레드의 상태를 복원하여 실행 대상을 변경하는 과정이다. 타이머 인터럽트로 할당된 실행 시간이 끝났을 때, 실행 중인 프로세스가 I/O를 요청해 대기 상태가 되었을 때, 더 높은 우선순위의 프로세스가 준비 상태가 되었을 때 발생할 수 있다. 컨텍스트 스위칭 자체는 프로그램의 실제 작업을 처리하지 않으므로 지나치게 자주 발생하면 성능이 저하된다.

참고 자료:

https://www.baeldung.com/cs/context-switching

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요.

답변:

* 프로세스 컨텍스트 스위칭에서는 CPU 레지스터 상태뿐 아니라 실행할 주소 공간도 변경해야 한다. 따라서 페이지 테이블과 같은 메모리 관리 정보가 바뀌며, TLB와 CPU 캐시에도 영향을 줄 수 있다. 
* 같은 프로세스에 속한 스레드 간 전환은 주소 공간과 대부분의 자원을 공유하므로 프로그램 카운터, 레지스터, 스택 등의 스레드 상태를 중심으로 전환한다. 일반적으로 같은 프로세스 내부의 스레드 전환이 프로세스 전환보다 비용이 작다.

참고 자료:

https://www.baeldung.com/cs/context-switching

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변:

유저 모드와 커널 모드는 프로그램이 수행할 수 있는 명령과 접근 가능한 자원을 제한하기 위해 구분한다. 일반 애플리케이션은 유저 모드에서 실행되며 하드웨어 직접 제어, 페이지 테이블 수정, 인터럽트 설정과 같은 특권 명령을 실행할 수 없다. 운영체제 커널은 커널 모드에서 실행되며 시스템 전체의 메모리와 장치에 접근할 수 있다. 이를 통해 사용자 프로그램의 오류나 악성 동작이 운영체제와 다른 프로세스에 직접적인 영향을 주는 것을 방지한다.

참고 자료:

https://www.baeldung.com/cs/user-kernel-modes

## [COMMON-012] 시스템 콜이 무엇이고, 사용자 프로그램이 시스템 콜을 호출하는 이유를 설명해 주세요.

답변:

시스템 콜은 유저 모드에서 실행되는 프로그램이 운영체제 커널의 기능을 요청하는 인터페이스이다. 파일 읽기와 쓰기, 프로세스 생성, 네트워크 통신, 메모리 관리 같은 작업은 보호된 자원에 접근해야 하므로 사용자 프로그램이 직접 수행할 수 없다. 따라서 프로그램은 시스템 콜을 통해 운영체제에 작업을 요청하고, 운영체제는 권한과 인자를 검사한 뒤 해당 작업을 대신 수행한다.

참고 자료:

https://www.geeksforgeeks.org/operating-systems/introduction-of-system-call/

## [COMMON-013] 시스템 콜이 실행될 때 유저 모드에서 커널 모드로 전환되는 과정을 설명해 주세요.

답변:

사용자 프로그램은 일반적으로 시스템 콜을 직접 호출하기보다 표준 라이브러리의 래퍼 함수를 호출한다. 래퍼 함수는 시스템 콜 번호와 인자를 정해진 레지스터에 저장한 뒤 `syscall`과 같은 특별한 명령어를 실행한다. CPU는 현재 실행 상태를 저장하고 권한 수준을 커널 모드로 변경한 뒤 운영체제의 시스템 콜 처리 코드로 이동한다. 커널이 요청을 처리하면 반환값을 설정하고 다시 유저 모드로 복귀하여 다음 명령어부터 실행한다.

참고 자료:

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.

답변:

인터럽트는 현재 실행 흐름을 잠시 중단하고 특정 사건을 우선 처리하도록 CPU에 알리는 신호이다. 인터럽트가 발생하면 CPU는 현재 프로그램 카운터와 레지스터 상태 등을 저장하고, 인터럽트 벡터를 통해 해당 인터럽트 처리 루틴으로 이동한다. 운영체제는 장치 상태 확인, 입력 데이터 처리, 타이머 갱신 등의 작업을 수행한다. 처리가 끝나면 저장했던 상태를 복원해 중단된 프로그램을 계속 실행하거나 스케줄러가 선택한 다른 작업을 실행한다.

참고 자료:

https://www.geeksforgeeks.org/operating-systems/interrupts/

## [COMMON-015] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이에 대해 설명해 주세요.

답변:

* 하드웨어 인터럽트는 키보드 입력, 디스크 I/O 완료, 네트워크 패킷 도착, 타이머 만료처럼 CPU 외부의 하드웨어 장치가 발생시키는 신호이다. 현재 실행 중인 명령어와 직접적인 관련 없이 비동기적으로 발생한다. 
* 소프트웨어 인터럽트는 프로그램이 특정 명령어를 실행하여 의도적으로 발생시키며, 전통적으로 시스템 콜을 요청하는 용도로 사용되었다. 즉 하드웨어 인터럽트는 외부 장치가 발생시키고, 소프트웨어 인터럽트는 실행 중인 프로그램에 의해 발생한다.

https://www.geeksforgeeks.org/computer-organization-architecture/interrupts-and-exceptions/

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변:

프로세스의 대표적인 상태는 생성, 준비, 실행, 대기, 종료 상태이다. 생성 상태는 운영체제가 프로세스를 만들고 필요한 자료구조와 자원을 준비하는 단계이다. 준비 상태는 CPU만 할당되면 실행할 수 있는 상태이고, 실행 상태는 실제로 CPU에서 명령어를 수행하는 상태이다. 대기 상태는 I/O 완료나 특정 이벤트를 기다리는 상태이며, 종료 상태는 실행을 마치고 운영체제가 관련 자원을 정리하는 상태이다.

참고 자료:

https://www.geeksforgeeks.org/operating-systems/states-of-a-process-in-operating-systems/

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변:

## [COMMON-026] Out-of-Order Execution이 무엇이고, 어떤 장점이 있는지 설명해 주세요.

## [COMMON-034] 멀티프로세스와 멀티스레드는 각각 어떤 상황에서 유리한지 설명해 주세요.

답변:

* 멀티프로세스는 각 작업이 독립적인 주소 공간을 사용하므로 하나의 프로세스에서 오류가 발생해도 다른 프로세스에 미치는 영향이 작다. 따라서 안정성과 격리가 중요하거나 서로 독립적인 CPU 연산을 병렬 처리할 때 유리하다. 
* 멀티스레드는 같은 프로세스의 메모리와 자원을 공유하므로 데이터 공유와 통신이 빠르고 생성 비용도 비교적 작다. 다만 공유 데이터에 대한 동기화가 필요하므로 데이터를 자주 공유하는 작업이나 하나의 서비스에서 여러 요청을 처리하는 상황에 적합하다.

## [COMMON-035] CPU-bound 작업과 I/O-bound 작업의 차이에 대해 설명해 주세요.

답변:

CPU-bound 작업은 실행 시간의 대부분을 계산에 사용하며 CPU의 연산 성능이 처리 속도를 결정하는 작업이다. 영상 처리, 암호화, 행렬 연산, 머신러닝 연산 등이 대표적이다. I/O-bound 작업은 디스크, 네트워크, 데이터베이스 등의 응답을 기다리는 시간이 긴 작업이다. CPU-bound 작업은 멀티코어 병렬 처리나 연산 최적화가 중요하고, I/O-bound 작업은 비동기 처리나 여러 스레드를 이용해 대기 시간 동안 다른 작업을 수행하는 것이 효과적이다.

## [COMMON-036] 인터럽트 방식과 폴링 방식의 차이에 대해 설명해 주세요.

답변:

* 폴링은 CPU가 장치의 상태를 일정한 주기로 반복해서 확인하는 방식이다. 구현이 단순하고 이벤트 발생 빈도가 매우 높은 상황에서는 효율적일 수 있지만, 이벤트가 없을 때도 계속 상태를 확인하므로 CPU 시간이 낭비될 수 있다. 
* 인터럽트 방식은 장치에서 처리할 사건이 발생했을 때 CPU에 신호를 보내는 방식이다. 불필요한 확인 작업을 줄일 수 있지만 이벤트가 지나치게 자주 발생하면 인터럽트 처리 비용이 커질 수 있다.

## [COMMON-037] DMA가 무엇이고, 왜 필요한지 설명해 주세요.

답변:

DMA는 Direct Memory Access의 약자로, 장치가 CPU를 거치지 않고 메모리와 직접 데이터를 전송할 수 있도록 하는 방식이다. DMA를 사용하지 않으면 CPU가 장치의 데이터를 하나씩 읽어 메모리에 복사해야 하므로 대용량 데이터 전송 중 CPU가 계속 사용된다. DMA를 사용하면 CPU는 전송 위치와 크기 등을 설정한 뒤 다른 작업을 수행할 수 있다. 전송이 완료되면 DMA 컨트롤러가 인터럽트를 통해 CPU에 결과를 알린다.

## [COMMON-038] 운영체제가 하드웨어 자원을 추상화한다는 말의 의미를 설명해 주세요.

답변:

운영체제가 하드웨어 자원을 추상화한다는 것은 애플리케이션이 복잡한 하드웨어 동작 방식을 직접 처리하지 않아도 되도록 일관된 인터페이스를 제공한다는 의미이다. CPU는 프로세스와 스레드라는 실행 단위로, 물리 메모리는 가상 메모리라는 독립적인 주소 공간으로 추상화된다. 저장장치와 여러 장치는 파일이라는 형태로 접근할 수 있고, 네트워크 통신은 소켓을 통해 처리할 수 있다. 이를 통해 프로그램은 하드웨어의 세부 구현을 알지 못해도 운영체제가 제공하는 인터페이스를 사용해 자원에 접근할 수 있다.

## [COMMON-039] 부팅 과정에서 운영체제가 메모리에 올라오는 흐름을 간단히 설명해 주세요.

답변:

컴퓨터의 전원이 켜지면 BIOS 또는 UEFI 펌웨어가 CPU와 메모리 등의 기본 하드웨어를 초기화하고 부팅 가능한 장치를 찾는다. 펌웨어가 부트로더를 실행하면 부트로더는 디스크에 저장된 운영체제 커널과 초기 RAM 파일 시스템을 메모리에 올린다. 커널은 메모리 관리, 인터럽트, 장치 드라이버, 스케줄러 등을 초기화하고 루트 파일 시스템을 연결한다. 이후 Linux에서는 `systemd`와 같은 최초의 사용자 공간 프로세스를 실행하고, 이 프로세스가 각종 시스템 서비스와 로그인 환경을 시작한다.

https://man7.org/linux/man-pages/man7/bootup.7.html

https://www.gnu.org/software/grub/manual/grub/grub.html

## [COMMON-040] 프로그램 실행 중 예외와 인터럽트는 어떤 차이가 있는지 설명해 주세요.

답변:

예외는 현재 실행 중인 명령어로 인해 발생하는 동기적인 사건이다. 0으로 나누기, 잘못된 메모리 접근, 페이지 폴트, 정의되지 않은 명령어 실행 등이 대표적이다. 인터럽트는 타이머 만료나 I/O 완료처럼 주로 CPU 외부 장치에서 현재 명령어 실행과 관계없이 발생하는 비동기적인 사건이다. 둘 다 현재 실행 흐름을 중단하고 운영체제의 처리 루틴으로 이동하지만, 현재 명령어에 의해 발생했는지 외부 사건으로 발생했는지가 핵심적인 차이이다.
