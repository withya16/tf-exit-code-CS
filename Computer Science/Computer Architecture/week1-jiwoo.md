## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변: 

CPU는 컴퓨터의 핵심 연산 및 제어장치입니다. 프로그램의 명령어를 읽어 들여 해석하고 (명령어 인출/해독) 그에따라 연산을 수행하며 컴퓨터 내부의 여러 장치인 메모리, 입출력 장치들을 제어하는 역할을 합니다.

- ALU는 산술논리연산장치로, 산술연산(덧셈/뺄셈/곱셈)과 논리연산(AND/OR/NOT)을 수행합니다. 피연산자는 레지스터에서 받고 결과를 레지스터에 저장합니다.
- 제어장치는 PC가 가리키는 주소에 따라 제어장치가 명령어를 메모리에서 가져와 해석하고, 각 단계마다 필요한 제어신호를 발생시켜 레지스터/ALU/메모리 등이 실제로 동작하도록 조율합니다.
- 레지스터는 메모리(RAM)보다 훨씬 빠르지만 용량은 매우 작으며, 연산에 필요한 데이터나 연산 결과를 임시로 저장합니다. 레지스터의 종류에는 프로그래밍 카운터(PC, Programming Counter), 명령어 레지스터(IR, Instruction Register), 메모리 주소 레지스터(MAR, Memory Address Register), 메모리 버퍼 레지스터(MBR, Memory Buffer Register 등이 있습니다.

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변:  

명령어 사이클이란 CPU가 메모리에 저장된 명령어 하나를 처리하기 위해 반복적으로 거치는 과정으로, Fetch → Decode → Execute 세 단계로 이루어집니다.

    1. Fetch (인출): PC가 가리키는 메모리 주소에서 명령어를 읽어와 IR에 저장하고 명령어를 하나 가져오면 PC 값이 자동으로 다음 주소를 가지도록 증가합니다.

    2. Decode (해석): 제어장치가 IR에 저장된 명령어를 읽어, 연산코드(Opcode)와 피연산자(Operand)로 나누어 분석합니다. 어떤 연산인지(덧셈, 데이터 이동, 분기 등), 어떤 부품(ALU, 레지스터 등)이 필요한지 판단합니다.

    3. Execute (실행): 해석 결과에 따라 제어장치가 제어신호를 발생시켜, ALU가 연산을 수행하거나 레지스터/메모리에 값을 저장하는 등 실제 동작이 일어납니다.

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변: 

1. 컴파일: 사람이 작성한 소스코드(예: .c 파일)를 컴퓨터가 이해할 수 있는 기계어로 변환하는 단계입니다.
2. 링킹: 여러 개의 목적 파일과 라이브러리(printf 같은 표준 함수들)를 하나로 합쳐서 실행 파일을 만드는 단계입니다.
3. 로딩: 만들어진 실행 파일을 보조기억장치(하드디스크/SSD)에서 주기억장치(RAM)로 옮기는 단계입니다.
4. 실행: CPU가 메모리에 올라온 명령어를 하나씩 가져와 처리하는 단계로, 이때부터 앞서 설명한 명령어 사이클(Fetch-Decode-Execute)이 반복되는 것입니다.

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변:

- 컴파일 언어는 전체 소스 코드를 컴파일러를 통해 기계어로 변환한 후 실행하는 언어로, 이때 발생하는 기계어 코드는 직접 실행 가능한 형태로 컴퓨터의 CPU가 직접 이해할 수 있습니다. 
- 인터프리터 언어는 소스 코드를 한 줄씩 읽어가며 실행하는 방식을 취하며, 이 때문에 별도의 컴파일 과정이 필요하지 않아 스크립트 언어라고도 불립니다. 인터프리터는 프로그램을 실행시킬 때마다 소스 코드를 기계어로 번역하는 역할을 수행합니다. 

- 핵심 차이 요약

| 구분 | 컴파일 언어 | 인터프리터 언어 |
|---|---|---|
| 번역 방식 | 전체 코드를 한 번에 변환 | 한 줄씩 읽으며 실행 |
| 실행 속도 | 빠름 (미리 기계어로 변환) | 상대적으로 느림 |
| 오류 발견 시점 | 컴파일 시 사전 발견 가능 | 실행 중에만 발견 |
| 개발 속도 | 상대적으로 느림 | 빠름 (바로 실행해볼 수 있음) |
| 대표 언어 | C, C++, Java | Python, JavaScript, Ruby |

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변: 

- JIT(Just-In-Time) 컴파일은 프로그램이 실행되는 시점에 기계어로 번역하는 컴파일 기법입니다. 프로그램 실행 중 자주 사용되는 코드를 기계어로 변환하여 성능을 최적화합니다. 따라서, 프로그램 실행 중에 자주 쓰이는 코드만 골라 기계어로 변환/캐싱함으로써 인터프리터의 유연함과 정적 컴파일의 속도를 동시에 노리지만, 그 대가로 초기 구동 오버헤드와 추가 메모리/보안 리스크를 안고 갑니다.

## [COMMON-006] 프로그램과 프로세스의 차이에 대해 설명해 주세요.

답변: 

- 프로그램(Program): 디스크에 저장된 정적인 실행 파일로 정적 존재입니다. 아직 실행 안 된 명령어와 데이터의 집합입니다.
- 프로세스(Process): 프로그램이 메모리에 올라와 OS로부터 자원(CPU, 메모리, 파일)을 할당받아 실행 중인 동적 존재입니다. 

-> 프로그램은 디스크에 저장된 채 실행을 기다리는 정적인 명령어 묶음이고, 프로세스는 그 프로그램이 메모리에 올라와 CPU와 메모리 등 자원을 할당받아 실제로 동작하는 동적인 상태입니다

- 핵심 차이

| 구분 | 프로그램 | 프로세스 |
|---|---|---|
| 상태 | 정적(passive) | 동적(active) |
| 위치 | 디스크(보조기억장치) | 메모리(RAM) |
| 자원 사용 | 없음 | CPU, 메모리 등 시스템 자원 할당받음 |
| 개수 관계 | 1개 | 하나의 프로그램이 여러 프로세스로 실행될 수 있음 |

특히 마지막 항목이 재밌는 포인트인데, 크롬을 두 번 클릭해서 두 개의 창을 띄우면 프로그램은 하나지만 프로세스는 각각 별도로 생성됩니다. 즉 **프로그램 하나의 인스턴스가 프로세스**인 셈이에요.

- https://velog.io/@gparkkii/ProgramProcessThread

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

답변:

- 프로세스는 운영체제로부터 독립된 메모리 공간과 시스템 자원을 할당받는 실행 단위이고, 스레드는 그 프로세스 내부에서 실제 명령어를 실행하는 흐름의 단위입니다. 스레드는 프로세스가 할당받은 자원 중 Code, Data, Heap 영역은 다른 스레드와 공유하고, Stack 영역만 독립적으로 가집니다. 이 때문에 스레드 간 통신은 메모리를 직접 공유하므로 비용이 적고, 컨텍스트 스위칭도 Stack만 교체하면 되어 프로세스 간 전환보다 빠릅니다. 반면 프로세스는 서로 메모리를 공유하지 않기 때문에 안정성은 높지만, 통신하려면 IPC가 필요하고 컨텍스트 스위칭 비용도 더 큽니다.

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.

답변: 
- PCB는 프로세스 제어 블록으로, 특정한 프로세스를 관리하는 데 필요한 정보를 포함하는 운영체제 커널의 자료구조입니다. 운영체제가 프로세스를 제어하기 위해 그 프로세스의 상태정보를 저장해놓는 구조체인 것입니다. 
- 컨텍스트 스위칭이 일어날 때, PCB가 이전 작업에 대한 내용을 기억하고 있기에 프로세스를 재시작하거나 정보 손실 없이 재개하기 위해서 사용합니다.

## [COMMON-009] 컨텍스트 스위칭이 무엇이고, 언제 발생하는지 설명해 주세요.

답변: 컨텍스트 스위칭은 CPU가 실행 중인 프로세스(또는 스레드)의 상태를 PCB에 저장하고 다음 실행할 프로세스의 상태를 불러오는 과정으로, 스케줄링/인터럽트/시스템 콜(모드 전환) 시점에 발생하며 그 자체가 오버헤드이기 때문에 과도하게 발생하면 성능이 저하됩니다.

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요.

답변: 

- 둘 다 실행중인 작업을 멈추고 다른 작업의 상태를 불러오는 컨텍스트 스위칭이지만 교환해야 하는 범위가 다릅니다.
- 프로세스 컨텍스트 스위칭은 메모리 주소공간까지 통째로 교체하면서 TLB/캐시 무효화가 발생해 비용이 크지만, 스레드 컨텍스트 스위칭은 같은 프로세스의 메모리 공간을 공유한 채 Stack과 레지스터만 교체하면 되기 때문에 훨씬 가볍습니다.

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변:

- 유저 모드란, 일반 사용자 프로그램(응용프로그램)이 실행되는 모드로, 제한적인 명령만 수행할 수 있습니다. 따라서, 하드웨어 자원에 직접 접근할 수 없습니다
- 커널 모드란, 운영체제가 CPU의 제어권을 가지고 운영체제 코드를 실행하는 모드로, 모든 종류의 명령을 실행할 수 있고 하드웨어 자원에 직접 액세스할 수 있습니다.

- 유저 모드와 커널 모드를 구분하는 이유는 사용자 프로그램이 하드웨어나 시스템 핵심 자원에 직접 접근하지 못하게 막아, 오작동이나 악의적인 코드로부터 운영체제와 다른 프로그램들을 보호하기 위해서입니다.

## [COMMON-012] 시스템 콜이 무엇이고, 사용자 프로그램이 시스템 콜을 호출하는 이유를 설명해 주세요.

답변:
- 시스템 콜은 사용자 프로그램이 파일/메모리/하드웨어처럼 커널 권한이 필요한 작업을 안전하게 요청할 수 있게 해주는 인터페이스로, 응용 프로그램이 시스템 자원에 직접 접근하면 보안/안정성 문제가 생기기 때문에 반드시 이 통로를 거쳐야 합니다.

## [COMMON-013] 시스템 콜이 실행될 때 유저 모드에서 커널 모드로 전환되는 과정을 설명해 주세요.

답변:

```
유저 모드 실행 중
   ↓ 시스템 콜 호출 (트랩 발생)
   ↓ 현재 상태를 PCB에 저장
   ↓ 모드 비트 0으로 전환 (커널 모드 진입)
   ↓ 인터럽트 벡터 → 시스템 콜 테이블 참조
   ↓ 커널이 요청된 작업 수행
   ↓ 모드 비트 1로 전환 (유저 모드 복귀)
   ↓ PCB에서 상태 복원
유저 모드로 복귀해서 이어서 실행
```
- 시스템 콜은 소프트웨어 인터럽트(트랩)를 발생시켜 현재 상태를 PCB에 저장하고 모드 비트를 전환함으로써 커널 모드로 진입하고, 커널이 인터럽트 벡터를 통해 해당 서비스 루틴을 실행한 뒤 다시 상태를 복원하며 유저 모드로 돌아오는 과정을 거칩니다.

답변:

## [COMMON-015] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이에 대해 설명해 주세요.

답변:

- 하드웨어 인터럽트는 외부 장치나 CPU 내부 하드웨어가 실행 흐름과 무관하게 비동기적으로 발생시키는 신호이고, 소프트웨어 인터럽트(트랩)는 실행 중인 프로그램의 특정 명령어(시스템 콜, 0으로 나누기 등)로 인해 동기적으로 발생하는 신호입니다.

| 구분 | 하드웨어 인터럽트 | 소프트웨어 인터럽트 |
|------|-----------------|------------------|
| 발생 원인 | 외부 하드웨어 장치 | CPU 내부, 명령어 실행 중 |
| 발생 시점 | 비동기 (언제든 발생) | 동기 (특정 명령어 실행 시) |
| 예시 | 키보드 입력, 디스크 I/O 완료, 네트워크 패킷, 타이머 | 시스템 콜, 0으로 나누기, 페이지 폴트 |
| 전달 방법 | CPU의 IRQ 핀으로 전기 신호 | `int` 명령어 또는 CPU 내부 감지 |

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변:


```
         admit          dispatch
 New  ─────────→  Ready  ────────→  Running
                    ↑                  │
             I/O complete         I/O request / wait
             interrupt              event wait
                    │                  │
                    └───── Waiting ←───┘
                                       │
                                  exit │
                                       ↓
                                  Terminated
```

- 프로세스는 New, Ready, Running, Waiting(Blocked), Terminated 다섯 가지 상태를 가집니다.
New는 프로세스가 생성되는 중인 상태, Ready는 실행할 준비는 됐지만 CPU를 아직 할당받지 못해 대기 중인 상태, Running은 실제로 CPU를 점유해 명령어를 처리 중인 상태입니다. Waiting은 I/O 요청 등으로 CPU를 할당받아도 당장 실행할 수 없어 이벤트를 기다리는 상태이고, Terminated는 실행을 마치고 자원을 반납하는 상태입니다.
중요한 포인트는 상태 전이 방향인데, Running에서 시간 할당량을 다 쓰면 Ready로, I/O 요청이 발생하면 Waiting으로 갑니다. 그리고 Waiting에 있던 프로세스는 이벤트가 끝나도 바로 Running으로 가지 않고 반드시 Ready를 거쳐서 스케줄링을 기다립니다.

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변:
- CPU는 한 번에 하나의 프로세스만 실행할 수 있는데, 여러 프로세스가 동시에 돌아가는 것처럼 보이게 하려면 '언제 어떤 프로세스에 CPU를 줄지' 결정하는 주체가 필요합니다. 이 역할을 하는 게 CPU 스케줄러입니다. 목표는 CPU 이용률을 높이고, 처리량은 늘리고, 대기 시간과 응답 시간은 줄이는 것입니다.
대표적인 알고리즘으로는 네 가지가 있습니다.

    -  FCFS는 도착한 순서대로 처리하는 가장 단순한 방식이지만, 긴 작업이 먼저 오면 뒤의 짧은 작업들이 오래 기다리는 Convoy Effect가 발생합니다.
    - SJF는 실행 시간이 가장 짧은 작업을 먼저 처리해서 평균 대기 시간을 최소화하지만, 실행 시간을 미리 알아야 한다는 현실적 한계가 있습니다.
    - 우선순위 스케줄링은 우선순위가 높은 프로세스부터 처리하는데, 낮은 우선순위 프로세스가 계속 밀리는 기아 현상이 생길 수 있어 에이징으로 보완합니다.
    - 라운드 로빈은 동일한 타임 슬라이스를 순환하며 배분하는 선점형 방식으로, 응답 시간이 중요한 시분할 시스템에서 널리 씁니다.

| 알고리즘 | 방식 | 장점 | 단점 |
|---------|------|------|------|
| FCFS | 도착 순서대로 | 구현 간단, 공평 | Convoy Effect (짧은 프로세스가 긴 것 뒤에서 오래 기다림) |
| SJF | 짧은 CPU burst 먼저 (비선점) | 평균 대기 시간 최소 | Starvation, 정확한 burst 예측 불가 |
| SRTF | SJF의 선점형 버전 | 이론적 최적 | Starvation, 오버헤드 |
| Round Robin | 각 프로세스에 Time Quantum씩 순환 | 공평, 응답성 좋음 | Time Quantum 크기에 민감 |
| Priority | 우선순위 높은 것 먼저 | 중요 작업 빠르게 | Starvation (Aging으로 해결) |
| MLFQ | 여러 우선순위 큐, 동적 조정 | 현실적 최선 | 구현 복잡 |

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

답변:

- 선점형 스케줄링과 비선점형 스케줄링의 차이는 실행 중인 프로세스로부터 CPU를 강제로 뺏을 수 있느냐에 있습니다. 
- 비선점형은 프로세스가 CPU를 할당받으면 스스로 종료하거나 I/O 요청으로 자발적으로 중단할 때까지 계속 실행됩니다. 그래서 응답 시간을 예측하기 쉽고 컨텍스트 스위칭 오버헤드가 적지만, 처리 시간이 긴 프로세스가 뒤의 짧은 프로세스들을 오래 기다리게 만들 수 있다는 단점이 있습니다.
- 선점형은 운영체제가 실행 중인 프로세스라도 강제로 중단시키고 다른 프로세스에 CPU를 할당할 수 있는 방식입니다. 타임 슬라이스가 끝나거나 더 높은 우선순위 프로세스가 들어오면 즉시 교체됩니다. 응답성이 좋아서 우리가 쓰는 대화형 시스템 대부분이 이 방식을 쓰지만, 그만큼 컨텍스트 스위칭이 자주 일어나서 오버헤드가 커집니다.
- 대표적으로 FCFS와 SJF는 비선점형, 라운드 로빈과 SRTF는 선점형 알고리즘입니다.

| 구분 | 비선점형 | 선점형 |
|---|---|---|
| CPU 뺏기 가능 여부 | 불가능 (자발적 반납만) | 가능 (강제로 뺏을 수 있음) |
| 응답 시간 예측 | 쉬움 | 상대적으로 어려움 |
| 컨텍스트 스위칭 오버헤드 | 적음 | 큼 |
| 적합한 시스템 | 배치 시스템 | 대화형/시분할 시스템 |
| 대표 알고리즘 | FCFS, SJF | RR, SRTF |

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변:
- Round Robin은 각 프로세스에게 동일한 Time Quantum만큼 CPU를 할당하고, 시간이 끝나면 준비 큐의 뒤로 보내는 선점형 스케줄링입니다. 이때 Time Quantum의 크기에 따라 trade-off가 발생합니다.
- Time Quantum이 너무 크면, 각 프로세스가 CPU를 오래 붙잡고 있게 되어 사실상 FCFS와 비슷하게 동작합니다. 컨텍스트 스위칭 횟수는 줄어들어 오버헤드는 적어지지만, 응답 시간이 느려져서 Round Robin을 쓰는 원래 목적인 빠른 응답성을 잃게 됩니다.
- 반대로 Time Quantum이 너무 작으면, 응답 시간은 빨라지고 대기 시간도 개선되지만, 선점이 너무 자주 일어나서 컨텍스트 스위칭 오버헤드가 커집니다. 극단적으로 작으면 실제 프로세스 작업 시간보다 컨텍스트 스위칭에 드는 시간이 더 커지는 역효과가 날 수 있습니다.

| Time Quantum | 장점 | 단점 |
|---|---|---|
| 너무 큼 | 컨텍스트 스위칭 오버헤드 적음 | 응답성 저하, 사실상 FCFS와 유사해짐 |
| 너무 작음 | 응답성 좋고 대기 시간 개선 | 컨텍스트 스위칭 오버헤드가 지나치게 커짐 |



참고 자료:
- https://velog.io/@chappi/OS%EB%8A%94-%ED%95%A0%EA%BB%80%EB%8D%B0-%ED%95%B5%EC%8B%AC%EB%A7%8C-%ED%95%A9%EB%8B%88%EB%8B%A4.-6%ED%8E%B8-%EC%8A%A4%EC%BC%80%EC%A4%84%EB%A7%813-%EC%84%A0%EC%A0%90%ED%98%95-%EC%8A%A4%EC%BC%80%EC%A4%84%EB%A7%81-%EC%95%8C%EA%B3%A0%EB%A6%AC%EC%A6%98Round-Robin-SRT-%EC%9A%B0%EC%84%A0-%EC%88%9C%EC%9C%84-%EB%B0%A9%EC%8B%9D-Multilevel-Queue

## [COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.

답변:
- 한줄요약: 멀티태스킹은 CPU 스케줄러가 각 프로세스에게 짧은 시간만큼 CPU 사용 순서를 정해주고, 컨텍스트 스위칭이 그 전환을 사람이 인지할 수 없을 만큼 빠르게 수행함으로써, 실제로는 순차 실행인데도 동시 실행처럼 보이게 만드는 것입니다.

- CPU는 한 순간에 하나의 프로세스만 실행할 수 있지만, 사용자에게는 여러 프로그램이 동시에 돌아가는 것처럼 보입니다. 이게 가능한 이유는 CPU 스케줄링과 컨텍스트 스위칭이 매우 빠르게 반복되기 때문입니다.
CPU 스케줄러는 시분할 방식으로 각 프로세스에게 짧은 타임 슬라이스만큼만 CPU 사용 시간을 배정하고, 시간이 다 되거나 I/O 요청이 생기면 다음에 어떤 프로세스를 실행할지 결정합니다.
그 전환이 실제로 일어나는 과정이 컨텍스트 스위칭입니다. 현재 실행 중이던 프로세스의 레지스터, 프로그램 카운터 같은 상태를 PCB에 저장하고, 다음 프로세스의 상태를 CPU에 적재하는 방식으로 진행됩니다. 이 전환은 보통 수백 나노초에서 수 마이크로초 단위로 매우 짧게 일어나기 때문에, 사람은 이 전환을 전혀 인지하지 못하고 여러 작업이 동시에 처리되는 것처럼 느끼게 됩니다.
즉, 실제로는 스케줄러의 결정과 컨텍스트 스위칭이 초당 수백~수천 번 반복되면서 순차 실행을 동시 실행처럼 보이게 만드는 겁니다.
